# Multi-Agent 协作中状态传递机制分析

## 概述

Multi-agent 协作的核心问题之一是：**状态如何在 agent 之间传递？** 不同的传递机制适用于不同的场景，理解这些机制有助于设计健壮的多 agent 系统。

---

## 状态传递的四种主要模式

### 1. 内存/上下文传递（Message Passing）

**原理**：Agent 基于 LLM，每次调用本质上是把历史消息序列（Context）作为输入。状态就是对话历史的累积。

**流程**：
```
[User msg] → [Agent A turns] → [Agent B turns] → [Agent C turns]
     ↓              ↓                ↓
  初始消息    A的回复+A思考    B的回复+B思考+C思考
```

**适用场景**：
- 串行执行的 agent 链（Sequential Pipeline）
- 父子 agent 嵌套调用（如 Claude Code 的 planner → code-reviewer）
- 短上下文窗口内的协作任务

**优点**：
- 实现简单，无需额外基础设施
- 状态自动维护，顺序天然保证
- 天然支持有状态的对话

**缺点**：
- Context 窗口有上限，长任务会溢出
- 并行 agent 无法直接共享状态
- 子 agent 结果合并有截断风险

**代表框架**：Claude Code、LangGraph（MessageGraph）、CrewAI（Sequential）

---

### 2. 共享状态存储（Shared Storage）

**原理**：将状态外部化到独立的存储系统，agent 通过读写存储交互，而非直接传递消息。

**存储介质选择**：

| 存储类型 | 适用场景 | 典型用例 |
|---------|---------|---------|
| Redis | 低延迟、高并发 | 实时协作、缓存中间结果 |
| PostgreSQL | 强一致性、结构化数据 | 任务状态、用户会话 |
| 文件系统 (JSON/MD) | 简单场景、审计需求 | 报告生成、文件交接 |
| 消息队列 (Kafka/RabbitMQ) | 事件驱动、松耦合 | 异步任务、事件通知 |

**流程**：
```
Agent A ──写入──→ [Shared Storage] ──读取──→ Agent B
```

**适用场景**：
- 并行 agent 需要共享数据
- 状态需要跨会话持久化
- 需要审计或回放 agent 决策过程
- 多 agent 异步通信

**优点**：
- 支持并行协作
- 状态可持久化、可恢复
- 解耦 agent，容错性强

**缺点**：
- 引入额外基础设施复杂度
- 一致性保证需要额外设计
- 存储格式和访问模式需要约定

**代表框架**：AutoGen（通过 python 任；务集成）、Microsoft Copilot Stack

---

### 3. 集中式状态管理（Blackboard / State Machine）

**原理**：中心化的状态存储，所有 agent 从中读取、写入，状态变更通过事件触发。

**架构**：
```
         ┌─────────────────────────┐
         │    Centralized Store     │
         │  (Blackboard / State)    │
         └─────────────────────────┘
              ↑    ↑    ↑    ↑
              │    │    │    │
         [Agent A] [Agent B] [Agent C] ...
```

**适用场景**：
- 多个 agent 需要协作解决同一个问题
- 需要统一的视图来监控所有 agent 状态
- 任务分配和状态同步需要集中控制

**优点**：
- 全局视图，便于监控和调试
- 状态变更可集中触发副作用
- 天然支持动态任务分配

**缺点**：
- 单点瓶颈和单点故障
- 中心化存储的扩展性挑战
- 复杂的状态转换逻辑

**代表框架**：IBM Beehive、OpenDIAG

---

### 4. 文件交接媒介（File-based Handoff）

**原理**：Agent A 将结果写入文件，Agent B 通过读取文件获取上下文。状态通过文件路径引用传递。

**流程**：
```
Agent A 输出 → output_a.json → Agent B 读取作为上下文
```

**适用场景**：
- 大输出不适合放在 Context 中时
- 需要持久化中间结果供后续使用
- Agent 运行在不同机器或不同时间

**优点**：
- 无 Context 窗口限制
- 结果可独立查看和调试
- 天然支持断点续传

**缺点**：
- 文件格式耦合
- 追加修改困难（全量覆盖）
- 缺乏原子性保证

**代表框架**：Claude Code（当 agent 输出过长时）、Magentic-One

## Claude Code 特定机制：子 Agent 结果合并

### 合并流程

```
主上下文 ──────────────────────────────────────────────
                    Agent tool call
                         ↓
                  [子 agent 独立进程]
                         ↓
                  返回结果消息
                         ↓
主上下文 ◄──────────────────────────────────────────────
```

### 合并规则

1. **仅最终结果合并**：只有子 agent 最后一条消息（完整回复）会被 append 到主会话消息历史
2. **中间过程不可见**：子 agent 内部的 extended thinking 对主会话不可见
3. **Tool Calls 记录保留**：主会话保留完整的 tool calls 记录（可见的工具名和参数）
4. **顺序追加**：结果按调用顺序线性追加到主上下文

### 结果合并后的消息结构示例

```
# 主会话消息历史
[User: 实现登录功能]
[Assistant: 我来规划一下...]
[Tool: Agent(id=xxx, task=规划)]
[Assistant: 根据规划，我需要...]
[Tool: Agent(id=yyy, task=写代码)]
[Assistant: 代码已完成...]
```

### 关键风险：Context 窗口膨胀

- 子 agent 返回内容越长，消耗父会话 token 越多
- 多次嵌套调用时，Context 会快速膨胀
- **优化策略**：
  - 控制子 agent 输出粒度（拒绝超长输出）
  - 使用摘要而非完整结果合并
  - 及时压缩或清理中间状态

## 场景化选型建议

### 按协作模式选择

| 协作模式 | 推荐状态传递机制 | 原因 |
|---------|----------------|------|
| 父子串行调用 | 上下文传递 | 顺序天然保证，实现简单 |
| 并行独立任务 | 共享存储 / 文件 | 避免上下文重复，去耦合 |
| 树形层级调用 | 上下文传递 + 文件 | 顶层用上下文，底层用文件 |
| 事件驱动协作 | 集中式状态管理 | 事件触发，状态变更驱动 |

### 按上下文窗口选择

| 场景 | 推荐机制 |
|-----|---------|
| 短任务（<32k tokens） | 纯上下文传递 |
| 中等任务（32k-128k） | 上下文传递 + 关键结果外部化 |
| 长任务（>128k） | 共享存储 + 文件交接 |

### 按一致性需求选择

| 需求 | 推荐机制 |
|-----|---------|
| 最终一致性即可 | 消息队列、文件 |
| 强一致性 | 数据库（事务）、集中式存储 |
| 实时一致性 | Redis + Pub/Sub |

## 混合模式的最佳实践

实际系统中，往往需要组合多种机制：

```
┌─────────────────────────────────────────────────────┐
│                    主 Agent (Planner)               │
│                   [上下文传递状态]                   │
└─────────────────────────────────────────────────────┘
                        │
          ┌─────────────┼─────────────┐
          ↓             ↓             ↓
    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ Agent A  │  │ Agent B  │  │ Agent C  │
    │(写文件)  │  │(读文件)  │  │(共享Redis)│
    └──────────┘  └──────────┘  └──────────┘
          │             │             │
          └─────────────┼─────────────┘
                        ↓
              ┌──────────────────┐
              │   结果汇总写入   │
              │  [中央存储/文件]  │
              └──────────────────┘
```

**原则**：
1. **热路径用内存**：频繁交互的状态走上下文传递
2. **冷路径用存储**：持久化结果走外部存储
3. **状态与计算分离**：状态归状态，Agent 只负责计算和决策
4. **显式交接点**：文件/存储的写入和读取点要明确，避免隐式依赖

## 常见问题与解决

| 问题 | 原因 | 解决方案 |
|-----|------|---------|
| Context 溢出 | 子 agent 输出过大 | 摘要压缩、文件外化 |
| 状态不一致 | 并行写共享存储 | 悲观锁/乐观锁/最终一致性 |
| 丢失中间状态 | 仅保留最终结果 | 定期快照到外部存储 |
| 循环依赖 | Agent 间互相等待 | 超时机制 + 降级策略 |
| 调试困难 | 黑盒状态流转 | 结构化日志 + 状态追踪 |

---

## 总结

| 机制 | 一致性 | 延迟 | 复杂度 | 适用规模 |
|-----|-------|-----|-------|---------|
| 上下文传递 | 强 | 低 | 低 | 小规模、串行 |
| 共享存储 | 可配置 | 中 | 中 | 中等规模、并行 |
| 集中式状态管理 | 强 | 低 | 高 | 大规模、实时协作 |
| 文件交接 | 弱 | 高 | 低 | 超长输出、离线协作 |

**核心洞察**：没有银弹，实际系统往往是多种机制的组合。选择时优先考虑**数据流是否清晰**、**状态是否有明确的单一来源**，以及**失败恢复是否容易**。
