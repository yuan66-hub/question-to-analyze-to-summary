# 任务拆分粒度控制指南

## 什么是任务拆分粒度

任务拆分粒度指的是将大任务分解为小任务的细致程度。粒度控制是任务管理中最核心的权衡之一。

---

## 粒度太细的问题

### 1. 管理开销超过实际价值
- 每个子任务都需要跟踪、更新、验收
- 协调会议和同步成本增加
- 管理者陷入微观管理

### 2. 失去全局视野
- 团队成员只看到碎片，看不到整体目标
- 难以理解自己工作的意义和上下文
- 任务之间的依赖关系变得模糊

### 3. 进度感知失真
- "完成了50个任务但项目毫无进展"的假象
- 大量短小任务产生虚假成就感
- 无法判断真实的交付节奏

### 4. 上下文切换损耗
- 频繁切换任务导致思维切换成本
- 深度工作被打断
- 质量下降和创新减少

### 5. 验收标准稀释
- 每个任务都"完成了"但整体不工作
- 集成问题被拖延到后期
- 最终交付风险增加

---

## 粒度太粗的问题

### 1. 估算困难且不准
- "实现用户模块"这种任务无法估算
- 风险和复杂性被隐藏
- 排期常常严重偏差

### 2. 责任归属模糊
- 任务太大，没人愿意主动承担
- "这不是我的责任"现象
- 出现空白地带

### 3. 进度跟踪失效
- "完成了30%"——这个数字怎么算出来的？
- 阻塞无法早期发现
- 临近deadline才发现问题

### 4. 并行化困难
- 大任务难以拆分给多人同时处理
- 形成单点瓶颈
- 团队整体效率受限

### 5. 变更适应性差
- 需求变化时需要重新规划整个大任务
- 返工成本高
- 客户响应能力下降

---

## 合适的粒度特征

### 理想范围：1-3人天

| 角色 | 建议粒度 |
|------|----------|
| 高级工程师 | 2-3人天 |
| 中级工程师 | 1-2人天 |
| 团队协作任务 | 3-5人天 |

### 可验证的完成标准

好的任务应该能够明确回答：
- 怎么算完成了？
- 谁能验证？
- 什么条件就算"做完了"？

### 独立可测试

- 能在不依赖其他任务的情况下验证
- 有明确的输入和输出
- 边界情况清晰

### 上下文自包含

- 所需信息都在任务描述中
- 不需要频繁追问才能开始
- 阻塞概率低

---

## 实践检验标准

### 问自己这些问题

1. **价值检验**：这个粒度对跟踪进度有价值吗？
2. **可并行性**：这个任务能否被一个人独立完成？
3. **验收明确性**：完成后能否明确判断是否合格？
4. **风险暴露度**：如果这个任务失败，影响多大？
5. **上下文保留**：别人能根据描述理解这个任务吗？

### 反面模式（需避免）

```
❌ "实现功能X" — 太粗
❌ "写方法A" — 太细
❌ "修复bug" — 缺少验收标准
❌ "完成模块A、B、C" — 多个任务混在一起
```

### 正面模式（参考）

```
✓ "实现用户登录API端点，包含参数验证和错误处理"
✓ "创建用户表迁移脚本，包含索引和约束"
✓ "编写登录模块单元测试，覆盖率>80%"
```

---

## 团队规模与粒度调整

| 团队规模 | 粒度倾向 | 原因 |
|----------|----------|------|
| 小团队(2-5人) | 偏粗 | 沟通成本低，需要灵活性 |
| 中型团队(5-15人) | 中等 | 需要平衡跟踪和灵活性 |
| 大团队(15+人) | 偏细 | 需要明确的依赖管理和责任划分 |

---

## 总结

| 问题 | 根因 | 解决方案 |
|------|------|----------|
| 粒度太细 | 过度追求可控性 | 以2-3人天为基准，重视任务价值而非数量 |
| 粒度太粗 | 缺乏分解意愿/能力 | 练习拆解，引入外部review |
| 粒度不稳定 | 没有统一标准 | 建立团队共识，使用时间盒方法 |

**核心原则**：任务应该足够小以能被可靠估算和独立完成，但足够大以保持上下文完整性和团队自主性。

## AI 辅助任务拆分

### Prompt 模板：需求 → 任务列表

```markdown
你是一位经验丰富的技术项目经理。请将以下需求拆分为开发任务列表。

需求描述：
[在此填写需求]

拆分要求：
1. 每个任务粒度控制在 1-3 人天
2. 每个任务必须有明确的完成标准（Definition of Done）
3. 标注任务依赖关系（哪些任务必须在另一个完成后才能开始）
4. 标注任务类型：后端 / 前端 / 联调 / 测试 / 文档
5. 对技术风险高的任务标注 ⚠️ 并说明风险点

输出格式：
| 任务 | 类型 | 工作量 | 依赖 | 完成标准 | 风险 |
```

### AI 辅助拆分示例

**输入需求**：实现用户评论功能，包含发布、点赞、举报、楼中楼回复

**AI 输出（示例）**：

| 任务 | 类型 | 工作量 | 依赖 | 完成标准 |
|------|------|--------|------|---------|
| 评论数据库表设计 | 后端 | 0.5天 | - | 包含索引、外键约束，通过 DBA 审核 |
| 评论 CRUD API | 后端 | 1天 | 数据库表 | 接口文档完整，单测覆盖率 > 80% |
| 点赞 API（防重复） | 后端 | 1天 | 评论 API | 并发测试通过，幂等性验证 |
| 举报 API + 审核队列 | 后端 | 1.5天 | 评论 API | ⚠️ 需要确认审核流程 |
| 评论列表前端组件 | 前端 | 1天 | 评论 API | 分页加载，骨架屏，空状态 |
| 楼中楼回复展开 | 前端 | 1天 | 评论列表组件 | 展开/收起动画，最多 3 层嵌套 |
| 前后端联调 | 联调 | 0.5天 | 以上全部 | 所有功能在测试环境验证通过 |

## 优先级排序框架

### MoSCoW 方法

```
M - Must Have（必须有）
  本次迭代的核心功能，没有它产品无法交付
  例：用户能发布评论

S - Should Have（应该有）
  重要但非关键，可在下个迭代补充
  例：评论支持富文本格式

C - Could Have（可以有）
  锦上添花，有时间再做
  例：评论支持表情包

W - Won't Have（本次不做）
  明确放弃，避免范围蔓延
  例：评论支持语音输入
```

### ICE 评分（适合功能优先级排序）

```
Impact（影响力）：1-10，对业务目标的影响程度
Confidence（置信度）：1-10，对影响力判断的把握程度
Ease（容易度）：1-10，实现难度（越容易分越高）

ICE Score = Impact × Confidence × Ease

例：
评论点赞：Impact=7, Confidence=8, Ease=9 → ICE=504  ← 优先做
楼中楼回复：Impact=5, Confidence=6, Ease=4 → ICE=120  ← 往后排
```

### 故事点（Story Points）估算参考

| 故事点 | 对应复杂度 | 典型工作量 |
|--------|-----------|----------|
| 1 | 极简单，已有类似实现 | 2-4 小时 |
| 2 | 简单，方案清晰 | 半天 |
| 3 | 中等，有少量不确定性 | 1 天 |
| 5 | 较复杂，涉及多模块 | 2-3 天 |
| 8 | 复杂，需要技术探索 | 需要拆分！|
| 13+ | 需要进一步分析和拆分 | 不应出现在 sprint |
