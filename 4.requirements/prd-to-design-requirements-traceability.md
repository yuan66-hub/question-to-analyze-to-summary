## 核心问题：需求为什么会丢失

### 需求丢失的常见原因

| 丢失环节 | 原因 | 后果 |
|---------|------|------|
| PRD → 拆解 | 需求粒度不清晰 | 子任务遗漏 |
| 拆解 → Design | 架构设计时忽略非功能性需求 | 实现时发现架构不支持 |
| Design → 代码 | 设计文档与代码实现脱节 | 最终行为与需求不符 |
| 代码 → 验证 | 缺少需求对应的测试用例 | 功能未真正被验证 |

### 需求追踪漏斗

```
┌─────────────────────────────────────────────────────────────┐
│                         PRD                                  │
│                    (原始需求输入)                              │
└─────────────────────────────────────────────────────────────┘
                              ↓ 拆解
┌─────────────────────────────────────────────────────────────┐
│                    功能清单 / Task List                       │
│              (将 PRD 转化为可执行任务)                          │
└─────────────────────────────────────────────────────────────┘
                              ↓ 设计
┌─────────────────────────────────────────────────────────────┐
│                       Design.md                              │
│              (架构设计与接口定义)                               │
└─────────────────────────────────────────────────────────────┘
                              ↓ 实现
┌─────────────────────────────────────────────────────────────┐
│                        代码                                   │
│                    (具体实现)                                 │
└─────────────────────────────────────────────────────────────┘
                              ↓ 验证
┌─────────────────────────────────────────────────────────────┐
│                       测试用例                                 │
│                  (验证需求是否满足)                            │
└─────────────────────────────────────────────────────────────┘
```

## 需求追踪体系

### 双向需求追踪矩阵

```
需求 ID    │ PRD 描述        │ Task       │ Design      │ 代码实现    │ 测试用例
───────────┼────────────────┼────────────┼─────────────┼────────────┼──────────
REQ-001   │ 用户登录        │ TASK-001   │ DESIGN-001  │ auth.go    │ auth_test
REQ-002   │ 支付功能        │ TASK-002   │ DESIGN-002  │ payment.go │ payment_test
REQ-003   │ 消息通知        │ TASK-003   │ DESIGN-003  │ notify.go  │ notify_test
```

### 每层检查点

#### 1. PRD 层检查

```markdown
## PRD 检查清单

- [ ] 每个需求有唯一 ID
- [ ] 每个需求可测试（有明确的验收标准）
- [ ] 需求之间无矛盾
- [ ] 非功能性需求已标注（性能、安全、兼容性）
```

#### 2. Design.md 层检查

```markdown
## Design 检查清单

- [ ] 每个 Task 有对应的设计章节
- [ ] 设计覆盖所有 PRD 需求点
- [ ] 接口定义清晰（含输入输出、错误码）
- [ ] 数据流完整（从请求到响应）
- [ ] 边界条件已考虑
```

#### 3. 代码实现层检查

```markdown
## 代码检查清单

- [ ] 需求 ID 体现在代码注释或 commit message
- [ ] 每个设计点有对应实现
- [ ] 单元测试覆盖核心逻辑
- [ ] 集成测试验证端到端流程
```

## AI 自检机制

### AI 自检的工作原理

AI 自检是让 AI 在生成内容后，**再次调用 AI 来验证内容质量**的一种技术。

```
┌──────────────────┐         ┌──────────────────┐
│   第一次生成       │         │    第二次自检      │
│   (主 Agent)      │ ──────→ │    (评审 Agent)   │
└──────────────────┘         └──────────────────┘
         │                              │
         ↓                              ↓
   初始内容输出                    验证报告 + 反馈
```

### AI 自检的类型

#### 类型一：规则检查（Rule-based）

```markdown
## 规则检查 Prompt

你是一个代码审查员。请检查以下内容是否符合规则：

规则列表：
1. 所有公共函数必须有文档注释
2. 不能使用 console.log
3. 错误必须被正确处理

检查内容：
[代码片段]

返回：
- 违规列表（如果有）
- 违规位置（行号）
- 修复建议
```

#### 类型二：一致性检查（Consistency Check）

```markdown
## 一致性检查 Prompt

你是一个需求追踪专家。请验证：

PRD 需求：
[PRD 内容]

Design 文档：
[Design 内容]

检查点：
1. Design 是否覆盖所有 PRD 需求？
2. Design 实现路径是否与 PRD 一致？
3. 是否有遗漏的需求点？

返回：
- 覆盖度评分 (0-100%)
- 遗漏的需求点列表
- 具体不一致之处
```

#### 类型三：完整性检查（Completeness Check）

```markdown
## 完整性检查 Prompt

你是一个架构评审专家。请检查：

设计文档：
[Design 内容]

标准检查项：
1. 错误处理是否完整？
2. 超时和重试机制是否定义？
3. 日志和监控点是否覆盖？
4. 安全考量是否到位？

返回：
- 缺失项列表
- 风险评级（高/中/低）
- 建议补充内容
```

## AI 自检的具体实现

### 实现模式

#### 模式一：两阶段生成（Two-Pass Generation）

```
第一阶段：生成
┌────────────────────────────────────────────┐
│ Prompt: "生成 Design.md 初稿"              │
└────────────────────────────────────────────┘
                        ↓
┌────────────────────────────────────────────┐
│ Design.md v1                               │
└────────────────────────────────────────────┘
                        ↓
第二阶段：自检
┌────────────────────────────────────────────┐
│ Prompt: "审查 Design.md 是否完整覆盖 PRD"  │
└────────────────────────────────────────────┘
                        ↓
┌────────────────────────────────────────────┐
│ 审查报告 + 反馈                            │
└────────────────────────────────────────────┘
                        ↓
必要时迭代生成 v2, v3...
```

#### 模式二：嵌入式自检（Inline Self-Check）

在生成内容时，同时生成自检结果：

```markdown
## Design.md 内容

### 模块 A 设计
[设计内容]

<!-- 自检结果 -->
✅ 已覆盖 REQ-001, REQ-002
⚠️ 缺少 REQ-003 的错误处理设计
⚠️ 未考虑边界情况：空指针、并发

### 模块 B 设计
[设计内容]

<!-- 自检结果 -->
✅ 已覆盖 REQ-004, REQ-005
✅ 错误处理完整
```

#### 模式三：独立 Agent 审查（Dedicated Review Agent）

```typescript
// 伪代码示例
async function generateWithSelfCheck(prd: PRD): Promise<Design> {
  // 1. 生成初稿
  const draft = await agent.generate(`生成 Design: ${prd}`);

  // 2. 独立审查 Agent
  const review = await reviewAgent.review({
    subject: draft,
    criteria: [
      { type: 'coverage', reference: prd },
      { type: 'consistency', reference: prd },
      { type: 'completeness', checklist: standardChecklist }
    ]
  });

  // 3. 如果审查发现问题，迭代改进
  if (review.issues.length > 0) {
    const improved = await agent.refine(draft, review.issues);
    return improved;
  }

  return draft;
}
```

## 误判情况分析

### AI 自检的常见误判类型

#### 误判类型一：过度严格（False Positive）

**表现：** AI 认为有问题，但实际没问题

**示例：**
```
Design 提到 "使用缓存提升性能"
AI 误判：❌ "未说明缓存失效策略"

实际情况：缓存策略在另一章节详细说明
```

**原因：** 上下文窗口不足，AI 看不到完整文档

**应对措施：**
- 确保自检时提供完整上下文
- 使用更精确的检查范围
- 人工复核 AI 标记的"问题"

#### 误判类型二：过度宽松（False Negative）

**表现：** AI 认为没问题，但实际有问题

**示例：**
```
Design 提到 "支持高并发"
AI 判断：✅ "已考虑性能需求"

实际情况：只提到"高并发"三个字，没有具体技术方案
```

**原因：** AI 接受了模糊的描述，没有深究具体实现

**应对措施：**
- 提供具体的检查标准（而非开放性问题）
- 设置最低检查项数量
- 人工抽检关键章节

#### 误判类型三：模式匹配误判

**表现：** AI 根据关键词判断，忽略实际语义

**示例：**
```
Design 章节： "错误处理"  →  AI 认为已覆盖
实际内容：    "详见文档" （链接已失效）
```

**原因：** AI 看到"错误处理"关键词即认为满足

**应对措施：**
- 检查引用的有效性
- 要求提供具体实现细节
- 人工复核引用类内容

#### 误判类型四：上下文混淆

**表现：** AI 误将 A 需求的设计归属到 B 需求

**示例：**
```
REQ-001 的设计被误判为覆盖 REQ-002
因为两者描述相似，AI 混淆了
```

**原因：** 需求描述相似时 AI 区分能力下降

**应对措施：**
- 使用唯一需求 ID
- 需求描述避免相似表述
- 逐条独立检查

### 误判率控制策略

#### 策略一：交叉验证

```
自检 Agent A（覆盖度）    自检 Agent B（一致性）
        ↓                          ↓
        └────────┬─────────────────┘
                 ↓
           人工合并结果
```

#### 策略二：阈值触发

```markdown
## 自动阈值规则

if (覆盖度 < 80%) → 自动标记为"需要人工审查"
if (问题数量 > 5)  → 自动标记为"需要人工审查"
if (检测到模糊描述) → 自动标记为"需要人工审查"
```

#### 策略三：迭代确认

```
自检报告 → 开发者确认 → 有疑问则重新检查 → 无疑问则通过
```

## 最佳实践

### 需求不丢失的关键措施

#### 1. 需求编号体系

```markdown
## PRD 结构示例

# PRD-2024-001: 用户认证系统

## 功能需求
- REQ-001: 用户名密码登录
- REQ-002: 第三方登录（OAuth）
- REQ-003: 登录失败锁定

## 非功能需求
- PERF-001: 登录响应时间 < 500ms
- SEC-001: 密码加密存储
- SEC-002: 登录日志审计
```

#### 2. Design 文档模板

```markdown
# Design: [模块名称]

## 需求追踪

| 需求 ID | 需求描述 | 设计章节 |
|---------|---------|---------|
| REQ-001 | xxx     | 2.1     |

## 架构设计
[内容]

## 接口设计
[内容]

## 需求覆盖声明
本文档覆盖 REQ-001 至 REQ-003 的所有需求点。
```

#### 3. 自检触发时机

| 阶段 | 触发时机 | 检查重点 |
|------|---------|---------|
| PRD → Design | 设计初稿完成时 | 需求覆盖度 |
| Design → 代码 | 模块实现完成时 | 与设计一致性 |
| 代码 → 测试 | 测试用例编写时 | 测试覆盖度 |
| 测试 → 发布 | 代码审查时 | 端到端验证 |

#### 4. 人工审查节点

```
PRD ──→ Design ──→ 代码 ──→ 测试 ──→ 发布
  │        │         │        │        │
  └────────┴─────────┴────────┴────────┘
              人工审查节点
```

### AI 自检使用建议

1. **不要完全依赖 AI 自检** - AI 是辅助手段，人工判断不可替代
2. **明确检查标准** - 越具体的 Prompt，误判率越低
3. **关注高风险项** - 安全、隐私、关键路径必须人工复核
4. **记录误判案例** - 建立误判模式库，持续优化 Prompt
5. **保持文档更新** - 自检发现的问题应及时修复文档

## 总结

| 维度 | 关键点 |
|------|-------|
| **需求追踪** | 双向追溯、编号体系、检查点 |
| **AI 自检** | 两阶段生成、嵌入式自检、独立审查 |
| **误判控制** | 交叉验证、阈值触发、人工确认 |
| **最佳实践** | 明确标准、关注高风险、持续优化 |

**核心原则：** AI 自检是增强人工审查的工具，而非替代人工的手段。保持人工在关键节点的判断权，是避免需求丢失的根本保障。
