# Cursor vs Claude Code 检索方案对比分析

## 概述

| 维度 | Cursor | Claude Code |
|------|--------|-------------|
| **检索核心** | Embedding + AST结构 | Glob + Grep + Read |
| **向量模型** | Code-specific embedding | 可配置 (OpenAI/Gemini等) |
| **索引方式** | 实时索引代码库 | 无持续索引，按需搜索 |
| **代码理解** | Treesitter AST | 文件内容 + 正则 |
| **混合搜索** | Vector + 结构 + 精确 | 纯文本搜索为主 |
| **上下文上限** | 动态窗口 | Token Budget控制 |
| **更新机制** | 文件变化触发重索引 | 每次命令重新读取 |

---

## 1. Cursor 检索方案

### 1.1 核心架构

```
用户查询/光标位置
       ↓
   ┌───┴───┐
   ↓       ↓
代码结构   语义向量
分析       检索
   ↓       ↓
   └───┬───┘
       ↓
   分数融合 → 重排 → 上下文注入
```

### 1.2 检索层级

| 层级 | 技术 | 说明 |
|------|------|------|
| **精确匹配** | substring/fuzzy search | 变量名、函数名、ID |
| **结构感知** | AST/Treesitter | 代码块、作用域 |
| **语义检索** | Embedding | 跨文件语义理解 |
| **混合** | 上述融合 | 综合评分 |

### 1.3 Cursor Composer 索引策略

```
代码库
    ↓
Treesitter解析 → AST结构
    ↓
分块策略:
├── 函数级块 (一个函数 = 一个块)
├── 类级块 (类定义 + 方法)
└── 文件级块 (小文件整体)
    ↓
Embedding向量化 (code-specific模型)
    ↓
HNSW索引
```

### 1.4 Cursor的独特设计

**Context Pool (上下文池)**:
- 不是把所有相关代码都塞进去
- 维护一个滑动窗口的候选上下文池
- 根据与当前光标的**距离**和**相关性**动态调整

**Prefix/suffix aware (前后缀感知)**:
- 理解当前编辑位置之前的代码 (prefix)
- 理解即将编写的代码结构 (suffix/模板)
- 检索时考虑这个双向上下文

### 1.5 Cursor优势场景

```
✅ 代码补全 (理解当前光标前后缀)
✅ 跨文件语义跳转 (如 "这个函数在哪里被调用")
✅ 基于注释生成代码
✅ 代码重构 (理解AST结构)
```

**原因**: Cursor对代码结构有深层理解，Embedding空间里保存了代码的语义+结构双重信息

## 2. Claude Code 检索方案

### 2.1 核心架构

```
用户请求
    ↓
解析WORKSPACE目录结构
    ↓
┌─────────────────────────────────┐
│  1. Glob模式匹配 (文件选择)      │
│  2. Grep全文搜索 (内容匹配)      │
│  3. 文件内容读取 (实际上下文)     │
└─────────────────────────────────┘
    ↓
Context压缩 (如果超限)
    ↓
组装到Prompt
```

### 2.2 检索策略

| 工具 | 用途 | 特点 |
|------|------|------|
| **Glob** | 文件路径匹配 | `**/*.ts`, `src/**` |
| **Grep** | 文本内容搜索 | 正则、语境行 |
| **Read** | 文件内容读取 | 行范围指定 |
| **Agent** | 主动探索代码库 | 递归搜索 |

### 2.3 Context管理机制

```
System Prompt
  ├── Tools list + schemas
  ├── Skills list (metadata only)
  ├── Workspace bootstrap files
  │   ├── AGENTS.md
  │   ├── SOUL.md
  │   ├── TOOLS.md
  │   └── ...
  ├── Conversation History
  └── Memory Search Results (注入)
```

### 2.4 上下文压缩策略

```
原始上下文 (可能很大)
    ↓
如果超过 token limit:
┌─────────────────────────┐
│  1. 摘要压缩 (Summarize)  │
│  2. 关键信息提取          │
│  3. 层级化组织            │
└─────────────────────────┘
    ↓
压缩后上下文 → 塞入模型窗口
```

### 2.5 Claude Code优势场景

```
✅ 精确文本搜索 (日志、配置、字符串)
✅ 大型代码库探索 (按目录结构)
✅ 一次性任务 (不需要持续索引)
✅ 与现有工具链集成 (git, shell)
```

**原因**: 基于文件的检索更可控、可解释，适合需要精确控制的场景

## 3. 两者融合趋势 - 现代Agent检索架构

```
┌─────────────────────────────────────────────┐
│           现代Agent检索架构                   │
├─────────────────────────────────────────────┤
│  1. 混合搜索 (Hybrid Search)                │
│     ├── 向量检索 (语义相似)                  │
│     └── BM25/关键词 (精确匹配)               │
│                                             │
│  2. 去重 (MMR - Maximal Marginal Releance) │
│                                             │
│  3. 重排 (Cross-Encoder Reranker)           │
│                                             │
│  4. 时间衰减 (Temporal Decay)               │
│     (近期内容权重更高)                        │
│                                             │
│  5. 可插拔Context Engine                    │
│     (支持自定义检索策略)                      │
└─────────────────────────────────────────────┘
```

### 代表项目: OpenClaw Memory

| 特性 | 标准 RAG | OpenClaw Memory |
|------|---------|-----------------|
| 存储格式 | 通常 Vector DB | **Markdown 文件** (SSOT) |
| 向量存储 | 专用向量数据库 | **SQLite + sqlite-vec** |
| 检索方式 | 纯向量检索 | **混合搜索** (Vector + BM25) |
| 后处理 | 简单 | **MMR + 时间衰减** |
| Context Engine | 固定 | **可插拔** |

---

## 4. 总结

| 工具 | 检索范式 | 核心创新 |
|------|---------|---------|
| **Cursor** | Embedding-first + AST结构 | 代码感知窗口 |
| **Claude Code** | Filesystem-first + 工具调用 | 可解释性 + 可控性 |
| **OpenClaw** | 混合检索 + MMR + 时间衰减 | 生产级RAG架构 |

**一句话概括**:
- Cursor像**"语义优先的代码搜索引擎"**
- Claude Code像**"文件系统操作的智能包装器"**
