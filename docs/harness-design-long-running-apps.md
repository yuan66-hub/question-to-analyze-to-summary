# 长时运行应用开发的 Harness 设计

> 原文：[Harness Design for Long-Running Application Development](https://www.anthropic.com/engineering/harness-design-long-running-apps)
> 作者：Prithvi Rajasekaran (Anthropic Labs)
> 发布时间：2026-03-24

## 核心问题

作者同时攻克两个关联难题：让 Claude 产出高质量前端设计，以及让它在无人干预下构建完整应用。单纯的提示词工程和简单 harness 设计都遇到了天花板，需要更新颖的 AI 工程方法来突破。

## 朴素实现为何失败

### 失败模式一：上下文连贯性丧失

- 随着上下文窗口填满，模型逐渐失去连贯性
- 部分模型（如 Sonnet 4.5）表现出 **"上下文焦虑"（context anxiety）**：在接近自认为的上下文极限时，提前草草收工
- **解决方案**：上下文重置（Context Reset）——彻底清空上下文窗口，启动全新 agent，通过结构化交接文件传递前一个 agent 的状态和后续步骤
- 这与 compaction（压缩摘要历史对话）不同——compaction 保留连续性但无法消除上下文焦虑，而重置提供干净的起点

### 失败模式二：自我评估失真

- Agent 评估自己的产出时，倾向于自信地给出好评——即使产出质量明显平庸
- 在设计等主观任务中尤为严重（没有类似测试通过/失败的二元验证）
- **解决方案**：将"做事的 agent"与"评判的 agent"分离——受 GAN（生成对抗网络）启发的多 agent 结构
- 独立的评估器更容易被调教为严格/怀疑态度，而让生成器自我批评则困难得多

## 前端设计：让主观质量可评分

### 四项评分标准

| 标准 | 含义 | 权重 |
|------|------|------|
| **设计质量（Design Quality）** | 设计是否是一个有机整体而非零件拼凑？颜色、排版、布局、图像是否共同营造出独特的氛围和调性？ | **高** |
| **原创性（Originality）** | 是否有定制化的设计决策？还是模板布局、库默认样式、AI 生成套路（如紫色渐变+白色卡片）？ | **高** |
| **工艺（Craft）** | 技术执行层面：排版层级、间距一致性、色彩和谐、对比度等基本功 | 低 |
| **功能性（Functionality）** | 独立于美学的可用性：用户能否理解界面、找到操作入口、完成任务 | 低 |

设计质量和原创性权重更高——因为 Claude 在工艺和功能性上本来就表现不错，但在设计感和原创性上经常产出平庸的"AI 味"作品。

### 关键工程实践

- **Few-shot 校准**：用带详细评分拆解的示例校准评估器，确保其判断与作者偏好对齐，减少跨迭代的评分漂移
- **评估器使用 Playwright MCP**：评估器不是看静态截图，而是通过 Playwright 实际导航页面、交互、截屏后再打分
- **每次生成 5-15 轮迭代**，整体运行长达 4 小时
- **生成器策略选择**：每轮评估后，生成器判断是继续优化当前方向还是彻底转换美学路线
- **措辞影响产出**：如"最好的设计应该是博物馆级别的"这类表述直接影响了生成器的风格走向

### 典型案例：荷兰艺术博物馆网站

前 9 轮迭代产出了干净的深色主题着陆页——技术过关但在预期之内。第 10 轮，生成器彻底推翻方案，重新想象为空间体验：CSS 透视渲染的 3D 房间、棋盘格地面、自由排列在墙上的画作、以门廊导航替代滚动或点击——这种创造性跳跃在单次生成中从未见过。

## 全栈编码：三 Agent 架构

### 架构设计

| Agent | 职责 |
|-------|------|
| **Planner（规划器）** | 将 1-4 句话的提示扩展为完整产品规格；刻意保持高层次，避免过早规定技术细节导致错误级联 |
| **Generator（生成器）** | 按 Sprint 逐个功能实现（React + Vite + FastAPI + SQLite/PostgreSQL）；每个 Sprint 结束后自评再交给 QA |
| **Evaluator（评估器）** | 用 Playwright MCP 像真实用户一样点击运行中的应用；按 Sprint 合约中的硬性阈值逐条评分 |

### Sprint 合约机制

- 在每个 Sprint 开始前，生成器和评估器**协商一份 Sprint 合约**：明确"完成"的定义
- 弥合了高层产品规格与可测试实现之间的鸿沟
- 生成器提出要构建什么以及如何验证成功，评估器审查确保方向正确，双方迭代直到达成共识
- **通信方式**：通过文件——一个 agent 写文件，另一个读取并在同一文件中回复或创建新文件

### 评估器捕获的真实 Bug 示例

| 合约标准 | 评估器发现 |
|----------|-----------|
| 矩形填充工具应支持点击拖拽填充 | **失败** — 工具仅在拖拽起止点放置瓦片，未正确触发 fillRectangle |
| 用户可选择并删除实体刷出点 | **失败** — Delete 键处理器要求同时设置 selection 和 selectedEntityId，但点击实体只设置了后者 |
| 用户可通过 API 重新排序动画帧 | **失败** — PUT /frames/reorder 路由定义在 /{frame_id} 之后，FastAPI 将 'reorder' 匹配为 frame_id 整数，返回 422 |

### 评估器调优经验

开箱即用的 Claude 是一个糟糕的 QA agent：
- 发现了合理的问题，然后说服自己这不是大问题并通过验收
- 倾向于表面测试，不深入边界情况

**调优方法**：阅读评估器日志 → 找到其判断与作者判断不一致的地方 → 更新 QA 提示词 → 多轮迭代直到评分合理。

## 实验对比

### 复古游戏制作器（Opus 4.5）

| Harness 类型 | 时长 | 成本 |
|-------------|------|------|
| 单 Agent（Solo） | 20 分钟 | $9 |
| 完整 Harness | 6 小时 | $200 |

**Solo 产出**：界面可用但游戏核心功能不工作（实体不响应输入），布局浪费空间，工作流僵硬。

**完整 Harness 产出**：16 个功能横跨 10 个 Sprint；包含精灵动画系统、行为模板、音效音乐、AI 辅助精灵生成和关卡设计、游戏导出分享链接；UI 更精致、游戏可以实际运行。

## 迭代简化：适配 Opus 4.6

### 核心原则

> Harness 中的每个组件都编码了一个关于"模型无法独立完成什么"的假设——这些假设值得压力测试，因为它们可能是错误的，也会随模型进步而过时。

### 具体简化

| 变更 | 原因 |
|------|------|
| **移除 Sprint 结构** | Opus 4.6 能原生处理长时连贯会话，不需要分块分解 |
| **评估器改为单次终评** | 而非每个 Sprint 评审 |
| **移除上下文重置** | Opus 4.6 不再表现出上下文焦虑 |
| **使用 SDK 自动 compaction** | 替代手动上下文管理 |

### DAW（数字音频工作站）实验结果

| Agent 阶段 | 时长 | 成本 |
|-----------|------|------|
| Planner | 4.7 min | $0.46 |
| Build（第 1 轮） | 2 hr 7 min | $71.08 |
| QA（第 1 轮） | 8.8 min | $3.24 |
| Build（第 2 轮） | 1 hr 2 min | $36.89 |
| QA（第 2 轮） | 6.8 min | $3.09 |
| Build（第 3 轮） | 10.9 min | $5.88 |
| QA（第 3 轮） | 9.6 min | $4.06 |
| **总计** | **3 hr 50 min** | **$124.70** |

- 生成器在无 Sprint 分解的情况下连续连贯运行超过 2 小时
- QA 仍然发现了真实问题：桩代码功能（如录音按钮只切换状态不捕获麦克风）、缺失交互（音频片段不能拖拽、不能分割）、效果可视化仅为数值滑块而非图形化界面
- 最终产出：浏览器内可运行的 DAW，包含编排视图、混音器、传输控制；内置 Claude agent 可通过提示词驱动整个音乐制作流程

## 关键经验总结

### 1. 实验驱动的调优

在目标模型上跑真实任务、阅读执行日志、针对性调优——没有捷径。

### 2. 评估器的价值是动态的

评估器在任务难度**超出模型独立可靠完成的边界**时才值得投入成本。随着模型能力提升，这个边界外移，评估器的适用范围也随之变化。

### 3. 任务分解按需使用

对弱模型（如 Sonnet 4.5），Sprint 分解是必需品；对强模型（如 Opus 4.6），可以完全去掉。

### 4. Harness 需要随模型迭代

新模型发布时：
- 剥离不再承载性能的组件
- 添加新组件以实现之前不可能的能力
- 重新校准评估标准

### 5. 有趣的 Harness 组合空间不会缩小——只会迁移

> "随着模型进步，有趣的 harness 组合空间不是在缩小，而是在移动。AI 工程师的有意义工作是不断找到下一个新颖的组合。"

## 可迁移的设计模式

```
┌─────────────────────────────────────────┐
│              用户提示词                    │
│          (1-4 句话描述)                   │
└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│           Planner Agent                  │
│  • 扩展为完整产品规格                      │
│  • 保持高层次，不规定技术细节                │
│  • 主动融入 AI 功能                       │
└───────────────┬─────────────────────────┘
                │ 产品规格文件
                ▼
┌─────────────────────────────────────────┐
│         Generator Agent                  │
│  • 逐功能实现                             │
│  • 与 Evaluator 协商 Sprint 合约          │
│  • 根据评估反馈选择优化或推翻重来            │
│  • 使用 git 做版本控制                     │
└───────┬─────────────────┬───────────────┘
        │ 代码 + 运行中应用  │
        ▼                 │
┌──────────────────┐      │
│  Evaluator Agent │      │
│  • Playwright    │      │
│    实际操作应用    │      │
│  • 按合约逐条评分  │──────┘
│  • 硬性阈值       │  反馈文件
│  • 详细 Bug 报告  │
└──────────────────┘
```
