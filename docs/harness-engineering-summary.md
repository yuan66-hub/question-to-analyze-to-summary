# Harness Engineering：从控制论看工程师角色的演变

## 核心论点

一个反复出现的控制论模式：**从亲手操作到设计控制系统**。

## 三次相同的模式

| 时代 | 之前 | 之后 |
|------|------|------|
| 1780s Watt 离心调速器 | 工人手动调阀门 | 飞球机构自动感知转速并调节 |
| Kubernetes | 工程师手动重启服务 | 声明期望状态，控制器自动协调 |
| 现在（Harness Engineering） | 工程师手写代码 | 设计环境与反馈回路，agent 写代码 |

每次模式出现，都是因为有人造出了足够强的**传感器+执行器**，在那个层级闭合了反馈回路。Wiener 1948 年命名了这个模式：控制论（cybernetics），与 Kubernetes 同源于希腊语"舵手"（κυβερνήτης）。

## 为什么代码库是最后的阵地

底层反馈回路早已存在——编译器管语法、测试管行为、linter 管风格。但架构判断层（这个改动是否契合系统设计？这个抽象长期是否合理？）一直没有传感器和执行器，只有人类能同时**感知**和**行动**。LLM 第一次在这个层级同时提供了两者。

## 校准是真正的难题

闭合回路是必要条件，不是充分条件。大多数人抱怨"agent 不理解我们的代码库"，但真正的问题是：**你脑子里的判断标准没有外化**。架构文档、定制 linter、黄金准则——这些把隐性知识变成机器可读的校准信息。OpenAI 每周五花 20% 时间清理"AI 垃圾"，直到把标准编码进 harness 本身。

## 不可逆的陷阱

文档、测试、架构约束——这些实践一直是正确的，只是以前偷懒的代价缓慢扩散。现在代价是即时且剧烈的：agent 以机器速度、全天候地忽略你的规范。更致命的是，**你无法用同样未校准的 agent 去收拾残局**。

## 一句话总结

工程师的工作从"写代码"变成了"设计让 agent 正确写代码的控制系统"，而这个控制系统的质量，取决于你能否把隐性的工程判断变成显式的、机器可读的规则。
