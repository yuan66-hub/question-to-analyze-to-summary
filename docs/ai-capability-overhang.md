# AI 的能力过剩（The Capability Overhang in AI）

> 来源：[Aaron Levie（Box CEO）- X 长文](https://x.com/levie/status/2037928306977509601)，2026-03-28
>
> "The future is already here — it's just not evenly distributed." — William Gibson

## 核心论点

AI 模型的能力已远超企业实际落地水平，存在巨大的**能力过剩（capability overhang）**。真正的瓶颈不在模型，而在于如何将 AI 能力嵌入企业工作流、连接真实数据与权限体系。

---

## 一、现状：编程领域爆发 vs 其他知识工作仍在早期

- 编码 Agent 已能独立完成大型项目，部分产品完全由 Agent 编写
- 科技行业之外的知识工作者中，AI 仍主要是"回答问题的助手"，大规模自动化处于萌芽阶段
- 前沿模型在法律、医疗、营销、写作等领域的知识日益深入，预计数年内在离散任务上将超越人类专家
- Ethan Mollick 观点：即使现在停止 AI 发展，当前技术也足以变革大量白领工作

## 二、Agent 落地的核心障碍——上下文鸿沟（Context Gap）

### 2.1 数据碎片化

编程之所以领先，是因为代码库天然包含 Agent 所需的全部上下文。而其他知识工作的关键信息分散在：

- 碎片化系统
- 视频会议 / 面对面会议
- 外部事件与客户动态

### 2.2 遗留系统

- 大量企业数据仍困在本地或老旧系统中
- 非结构化数据无法轻易被云端 Agent 访问
- 向云原生 / AI 原生架构迁移仍是前置依赖

### 2.3 权限与访问控制

- 组织中没有两个用户有完全相同的数据权限
- Agent 大规模执行工作时，权限管理复杂度呈指数级上升
- 有时 Agent 需要比用户更多的权限，有时需要更少——两者都难以管理

### 2.4 技术标准快速变化

18 个月内，企业可能依次经历以下架构范式的切换：

- 向量数据库 + RAG 检索
- MCP Server 连接各 SaaS 工具
- 代码执行 Agent 通过 CLI 操作平台

刚标准化一套方案就可能过时。

## 三、巨大的机会

### 3.1 软件产品

- 让非技术用户轻松管理 Agent 的产品将胜出
- 编码工具已从"辅助写代码"演变为"管理和监督 Agent 工作"，同样的转变将发生在法律、金融、咨询、制造等所有知识工作领域

### 3.2 平台格局

| 类型 | 机会 |
|------|------|
| 现有平台 | 深度绑定数据与工作流，通过上下文工程（context/harness engineering）端到端解决问题 |
| 新玩家 | 在垂直行业或业务线场景中重新定义工作流 |
| 水平层 | 跨系统的智能连接层 |

### 3.3 专业服务与变革管理

- 企业需要大量帮助来为 Agent 准备好环境，**FDE（Field Engineering）** 趋势将持续
- AI 公司需要深入理解客户工作流，帮助企业将 Agent 应用到具体流程
- 系统集成商和咨询公司如果能快速转型，将在 Agent 落地中获得巨大机会

### 3.4 企业内部人才

- 懂得为 Agent 重新设计工作流的人将极具价值
- 核心是将依赖"部落知识"或模拟方式协作的流程，重构为 Agent 可消费的形式

## 四、技术扩散需要时间

以云计算类比：

| 时间 | 云计算状态 |
|------|-----------|
| 2010 年 | 硅谷人人都知道云是未来；AWS 年收入约 5 亿美元；Azure 刚发布；GCP 还叫 Google App Engine |
| 2025 年 | 三大云平台年收入约 2250 亿美元（仅占云市场约 60%），增长近 **1000 倍** |

Agent 也不例外——扩散需要时间，但这正是机会所在。

## 五、总结

> AI 的能力已经足够强大，真正的战场不在模型本身，而在于谁能把 AI 能力与企业真实的数据、权限、工作流连接起来，让 Agent 真正落地到知识工作中。赢家将是那些能够弥合技术突破与真实世界运作方式之间鸿沟的人。

## 六、为什么 Agent 会“卡半天才吐字”

如果说前文讨论的是 Agent 为什么还没有大规模落地，那么在产品体验层面，一个高频问题是：**为什么 Agent 明明已经很强，却常常不像普通聊天那样立即流式输出？**

核心原因是，普通 Chat 的流式大多是“生成流式”，而 Agent 的流式本质上是“编排流式”。

### 6.1 真正慢的往往不是生成，而是首字之前

用户体感最敏感的指标不是 `tokens/sec`，而是 `TTFT`（Time to First Token，首字时间）。哪怕后续输出很快，只要第一句话来得太晚，用户就会觉得系统“卡住了”。

Agent 的首字时间更长，通常不是因为模型解码更慢，而是因为在开口前还要先完成一整套准备动作：

- 恢复会话与系统状态
- 组装历史对话、规则、当前文件、git 状态等上下文
- 判断任务类型，决定是否要调用工具
- 执行文件读取、搜索、命令、MCP 等操作
- 汇总结果后再组织回答

因此，很多“卡半天才吐字”的现象，本质上不是模型在慢慢想，而是系统还停留在**生成前的编排阶段**。

### 6.2 Agent 的“上下文鸿沟”也体现在交互体验里

前文提到，Agent 落地的核心障碍是上下文鸿沟（Context Gap）。这不仅影响企业部署，也直接影响响应体验。

编程场景之所以更容易做出高质量 Agent，是因为代码仓库天然提供了高度结构化、可机器读取的上下文；但即便如此，Coding Agent 在真正回答前，仍然往往需要读取文件、理解仓库结构、检查状态和规则。

换句话说，**上下文越复杂，Agent 开口越晚**。企业知识工作中的上下文分散问题，在线上产品里就表现为更高的首字延迟、更不稳定的流式节奏，以及用户对“它到底是不是卡了”的不确定感。

### 6.3 工具调用会打断连续流式

Agent 和普通聊天产品的另一个关键区别在于：前者往往不是“收到问题就直接回答”，而是先决定要不要使用工具。

一旦进入工具调用阶段，流式输出就会天然变成分段式：

1. 模型先做决策，判断下一步需要哪些工具
2. 系统执行搜索、读取、测试、浏览器或远端服务调用
3. 工具返回结果后，模型再进行综合并输出答案

这意味着很多时候用户看到的并不是连续不断的 token 流，而是：

`停顿 -> 出现一小段说明 -> 再停顿 -> 输出结果`

所以 Agent 的问题不只是“生成速度慢”，而是**决策、工具、等待、总结**共同构成了响应链路。

### 6.4 工程优化的重点已经变了

如果把 Agent 看作一个完整系统，那么优化重点就不该只放在模型吞吐，而应该放在端到端体验上：

- 优先降低 `TTFT`，而不是只提升 `tokens/sec`
- 减少无关上下文注入，控制 prompt 膨胀
- 对历史摘要、仓库索引、规则解析做缓存
- 将文件读取、搜索、工具调用尽量并行化
- 把“正在读取文件 / 正在搜索 / 正在运行测试”显式展示给用户
- 减少网关和前端渲染层的缓冲

这背后的产品原则是：**宁可先给用户一个可信的进度反馈，也不要沉默等待一个完美答案。**

### 6.5 一个更准确的理解框架

从工程视角看，可以把两类系统做一个简单区分：

| 类型 | 本质特征 | 主要瓶颈 |
|------|----------|----------|
| 普通 Chat | 生成流式 | 模型推理与解码速度 |
| Agent | 编排流式 | 上下文装配、工具决策、工具执行、结果汇总 |

因此，Agent 的“卡顿感”常常不是模型弱，而恰恰是因为系统在真正回答前做了更多事。随着 Agent 越来越深入真实工作流，这种现象不会自然消失，只会要求产品和架构更重视过程可见性、上下文工程与工具编排能力。
