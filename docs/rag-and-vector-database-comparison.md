# RAG 方案与向量数据库全景分析

## 一、主流 RAG 架构方案

### 1. Naive RAG（基础 RAG）

最基本的 "检索-生成" 管线：

```
文档 → 分块 → Embedding → 向量库 → 查询检索 Top-K → LLM 生成
```

**优点：** 实现简单，入门快
**缺点：** 召回质量有限，无法处理多跳推理、全局摘要等复杂场景

---

### 2. Advanced RAG（增强 RAG）

在 Naive RAG 基础上引入多个优化环节：

| 阶段 | 技术 | 说明 |
|------|------|------|
| **预处理** | Semantic Chunking / Parent-Child Chunking | 按语义边界切分，保留上下文 |
| **索引** | Hybrid Index (Dense + Sparse) | 向量 + BM25 关键词双路召回 |
| **查询** | Query Rewriting / HyDE / Query Decomposition | 改写或分解用户问题提升召回 |
| **检索后** | Reranker (Cohere Rerank, BGE-Reranker) | Cross-encoder 二次排序提升精度 |
| **生成** | Contextual Compression / Citation | 压缩上下文，附加引用来源 |

**适用场景：** 生产级问答系统、企业知识库

---

### 3. GraphRAG（微软提出）

```
文档 → 实体/关系抽取 → 知识图谱 → 社区检测 → 分层摘要 → 查询
```

**优点：**
- 支持全局摘要类问题（"整个数据集的主题是什么？"）
- 多跳推理能力强
- 实体关系可解释

**缺点：**
- 构建成本高（需要大量 LLM 调用来抽取实体）
- 图谱更新维护复杂
- 对小规模简单问答场景 overkill

**适用场景：** 法律/医疗文档分析、复杂报告生成、需要跨文档推理的场景

---

### 4. Agentic RAG

将 RAG 与 Agent 架构结合：

```
用户问题 → Agent 判断 → 选择工具(检索/SQL/API/计算) → 多轮推理 → 生成
```

**优点：**
- 动态决策检索策略
- 支持多数据源混合查询
- 可自我修正（检索不足时重新查询）

**缺点：**
- 延迟较高（多轮 LLM 调用）
- 调试复杂度高
- 成本较高

**适用场景：** 复杂业务助手、多源数据融合、需要动态决策的场景

---

### 5. RAPTOR（树状递归摘要）

```
文档分块 → 聚类 → 摘要 → 再聚类 → 再摘要 → 多层树结构
```

**优点：** 不同抽象层级的检索，兼顾细节和全局
**适用场景：** 长文档理解、多层次问答

---

## 二、主流 RAG 框架对比

| 框架 | 语言 | 特点 | 适用场景 |
|------|------|------|----------|
| **LlamaIndex** | Python/TS | 专为 RAG 设计，索引/检索抽象丰富 | RAG 优先项目，快速原型 |
| **LangChain** | Python/TS | 生态最大，Chain/Agent 灵活组合 | 复杂 Agent 管线，多工具编排 |
| **Haystack** | Python | 生产级管道，组件化设计 | 企业搜索，生产部署 |
| **RAGFlow** | Python | 深度文档理解（OCR、表格、版面） | 复杂文档格式（PDF/扫描件） |
| **Dify** | Python | 低代码平台，可视化编排 | 快速交付，非技术团队 |
| **Verba** | Python | Weaviate 深度集成 | 已选 Weaviate 的团队 |

### 逐一详细分析

#### 1. Milvus / Zilliz Cloud

**优点：**
- 真正的分布式架构，存算分离，支持百亿级向量
- GPU 索引加速（CAGRA、IVF-PQ）
- 丰富的索引类型（HNSW, IVF, DiskANN, SCANN）
- 成熟的多租户和 RBAC
- Zilliz Cloud 提供全托管服务

**缺点：**
- 部署复杂（依赖 etcd, MinIO/S3, Pulsar/Kafka）
- 运维成本高，学习曲线陡峭
- 对小规模场景过于笨重

**适用场景：**
- 超大规模生产系统（推荐、搜索、广告）
- 需要处理 10 亿+ 向量的场景
- 有专业运维团队的企业

---

#### 2. Pinecone

**优点：**
- 零运维，全托管 Serverless 架构
- 开发体验极佳，5 分钟上手
- 自动扩缩容，按量付费
- Serverless 模式大幅降低成本

**缺点：**
- 闭源，供应商锁定
- 数据离开自己的基础设施（合规风险）
- 高级功能（Metadata 过滤）在大规模下性能一般
- 自定义能力有限

**适用场景：**
- 快速验证 MVP / PoC
- 不想管基础设施的创业团队
- 中小规模（<1 亿向量）应用

---

#### 3. Weaviate

**优点：**
- 内置向量化模块（text2vec-openai, img2vec 等），无需外部 Embedding
- 原生混合搜索（BM25 + 向量），开箱即用
- GraphQL API 设计优雅
- 模块化架构，扩展性强
- 多模态支持好

**缺点：**
- 超大规模性能不如 Milvus
- Go 语言生态，二次开发需要 Go 基础
- 内存消耗较高

**适用场景：**
- 需要混合搜索的语义搜索系统
- 多模态检索（文本+图片）
- 希望减少外部依赖（内置向量化）

---

#### 4. Qdrant

**优点：**
- Rust 编写，性能极佳，内存效率高
- 丰富的过滤能力（payload filtering）
- 量化压缩优秀（Scalar / Product Quantization）
- API 设计简洁清晰
- 单机性能最强之一

**缺点：**
- 分布式能力相对 Milvus 较弱
- 生态和社区规模小于 Milvus/Weaviate
- 大规模集群部署经验较少

**适用场景：**
- 对延迟和性能要求极高的场景
- 需要复杂过滤条件的向量搜索
- 中等规模（千万到亿级）应用
- Rust 技术栈团队

---

#### 5. Chroma

**优点：**
- 极简 API，Python-native，嵌入式部署
- 与 LangChain / LlamaIndex 深度集成
- 本地开发零配置
- 学习成本最低

**缺点：**
- 不适合生产大规模部署
- 无分布式支持
- 缺乏高级功能（RBAC、备份、监控）
- 高并发下性能有限

**适用场景：**
- 本地开发和原型验证
- 小规模应用（<100 万向量）
- 教学和实验
- 嵌入式场景（桌面应用）

---

#### 6. pgvector

**优点：**
- 直接在 PostgreSQL 中使用，无需新增基础设施
- SQL 生态完整（JOIN、事务、备份全复用）
- HNSW 索引支持（0.5+ 版本大幅提升性能）
- 运维成本最低（复用现有 PG 运维体系）
- 强一致性保证

**缺点：**
- 非专用向量数据库，大规模性能天花板
- 缺乏高级向量搜索功能（量化、GPU 加速）
- 单机扩展受限
- 高维向量（>1536 维）性能下降明显

**适用场景：**
- 已有 PostgreSQL 的项目，想快速添加向量搜索
- 向量数据与业务数据需要强一致性
- 中小规模（<1000 万向量）
- 不想引入额外中间件

---

## 四、选型决策树

```
你的向量规模是多少？
├── < 100 万 → 原型/实验
│   ├── 已有 PostgreSQL → pgvector
│   └── 纯 Python 开发 → Chroma
│
├── 100 万 ~ 1 亿 → 中等规模生产
│   ├── 不想管运维 → Pinecone
│   ├── 需要混合搜索 → Weaviate
│   ├── 追求极致性能 → Qdrant
│   └── 已有 PG 且规模 <1000 万 → pgvector (HNSW)
│
└── > 1 亿 → 大规模生产
    ├── 有运维团队 → Milvus (自部署)
    └── 无运维团队 → Zilliz Cloud / Pinecone Enterprise
```

## 五、2025 年趋势总结

1. **混合搜索成为标配** — 纯向量搜索不够，Dense + Sparse + Reranker 三段式成为最佳实践
2. **pgvector 快速追赶** — HNSW 索引让它在中小规模场景下足够好用，避免引入新组件
3. **Agentic RAG 兴起** — RAG 从静态管线走向动态 Agent 决策
4. **GraphRAG 落地** — 知识图谱与向量检索融合，处理复杂推理场景
5. **Contextual Retrieval** — Anthropic 提出的在分块时嵌入上下文信息，提升检索质量
6. **Late Chunking / ColBERT** — 更细粒度的 token 级交互检索方式
