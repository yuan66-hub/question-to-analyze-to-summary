# 从 RAG 到 Knowledge OS：LLM Wiki 应该怎么工程化落地

## 一句话总结

很多团队理解 `LLM Wiki` 时，第一反应是做一个 `RAG + 文档站 + 聊天框`。

这条路可以很快做出 demo，但一旦系统开始持续接入聊天、工单、PR、Agent 轨迹、运行日志，并尝试自动沉淀知识，问题就会立刻从“检索准不准”升级成“知识会不会越来越脏”。

如果压缩成一句工程表达：

`LLM Wiki = retrieval + evidence management + knowledge governance + controlled publishing`

也就是说，一个能真正落地的 `LLM Wiki`，本质上不是“会回答问题的文档站”，而更像一个统一知识中台，或者说一个轻量版的 `Knowledge OS`。

---

## 1. 为什么普通 RAG 很难撑起真正的 LLM Wiki

很多团队在做知识产品时，最自然的路径是：

1. 把文档切 chunk
2. 做 embedding
3. 用向量库检索
4. 把命中的片段拼进 prompt
5. 再套一个聊天 UI

demo 阶段，这套方案通常已经足够。

但只要你希望这个系统不只是“回答问题”，而是开始承担下面这些职责，普通 RAG 很快就会暴露结构性短板：

- 接入多种知识源，而不只是 Markdown 文档
- 让 Agent 从执行过程里学习经验
- 自动整理 FAQ、SOP、故障排查卡片
- 把新知识持续写回系统
- 在冲突、过时、撤回、权限变更时仍然保持稳定

这时候你会发现，问题早就不是“召回 Top-K 的效果怎么样”了，而是：

### 1.1 你检索到的是内容，不一定是知识

工单评论、会议纪要、聊天记录、Agent trace 都可以被检索，但这些内容往往是：

- 局部的
- 临时的
- 观点性的
- 上下文依赖强的
- 权威性不一致的

它们适合做“证据候选”，不适合直接变成“正式知识”。

### 1.2 你可以写回内容，但不等于能治理知识

很多系统一开始觉得：检索都做出来了，让 Agent 再自动生成 FAQ 页面不就行了？

问题在于，写回一旦开始，系统立刻会遇到这些难题：

- 新知识和旧知识是不是重复
- 新结论和旧结论冲不冲突
- 这条知识适用于哪个版本、哪个租户、哪个环境
- 它是稳定事实，还是一次性 workaround
- 上游来源被删除、撤权、修订后，下游页面怎么办

换句话说，**写回的难点不是生成，而是治理**。

### 1.3 你有搜索系统，不一定有发布系统

很多 RAG 产品只有“存”和“查”，没有真正意义上的“发布”。

但只要你开始做 `LLM Wiki`，就必须回答这些问题：

- 什么内容只能作为候选
- 什么内容可以进入正式页面
- 什么内容虽然还存在于系统里，但已经不能再被用户看到
- 什么内容应该降级、撤回、归档或重新编译

所以，从工程视角看，`LLM Wiki` 真正需要补齐的，不只是 retrieval，而是一个完整的知识生命周期。

## 2. LLM Wiki 不是“AI 文档站”，而是“知识加工流水线”

如果想把这件事讲清楚，我更倾向于把 `LLM Wiki` 理解成一条加工链，而不是一个聊天入口。

它的核心链路应该是：

`source -> evidence -> knowledge unit -> governance -> publish -> consume`

这 6 个环节里，真正决定系统上限的通常不是左边的检索，也不是右边的聊天，而是中间三层：

- `evidence`
- `knowledge unit`
- `governance`

### 2.1 Source：接入混合知识源

第一层是所有原始来源，例如：

- Markdown / Notion / PDF
- PR / issue / commit discussion
- 工单系统
- 聊天记录
- 会议纪要
- Agent trace
- tool result
- 运行日志

这一层解决的是“知识从哪里来”。

### 2.2 Evidence：把内容先沉淀成可追溯证据

很多系统的问题，是把原始内容一上来就当成可消费知识。

更稳的做法是先把内容变成 `Evidence`：

- 保留原始来源
- 保留时间、作者、租户、权限范围
- 保留片段边界
- 保留可追溯引用

这一步很重要，因为它决定了系统后面能不能做：

- 溯源
- 回放
- 撤回
- 权限传播
- 证据加权

如果没有证据层，后面的“知识治理”很容易退化成“模型看着办”。

### 2.3 Knowledge Unit：从内容里提炼可治理的知识单元

证据不是最终形态。真正适合被系统治理和发布的，是结构化知识单元。

第一版通常不需要支持太多类型，4 到 6 类就够了，例如：

- `fact`
- `procedure`
- `faq`
- `failure_pattern`
- `decision`
- `constraint`

为什么要先抽成知识单元？因为这样系统才有机会回答这些问题：

- 这是不是和已有知识是同一件事
- 它是不是一个新版本
- 它适用于哪些 scope
- 它现在该不该被发布
- 它和哪些知识冲突、依赖、细化或替代

如果没有这一层，你就只有一堆 chunk，而没有真正可治理的知识对象。

### 2.4 Governance：让系统知道什么能留下，什么不能

这是我认为整个 `LLM Wiki` 最难、但也是最容易被低估的一层。

治理层至少要做这几件事：

- 去重
- 冲突检测
- 可信度打分
- 时效性衰减
- 审查与升级
- 版本推进
- ACL / 权限传播
- 撤回与回滚

很多系统失败，不是因为 retriever 不够强，而是因为没有治理层，最后知识库越跑越像一个“自动堆文档机”。

### 2.5 Publish：把治理后的知识编译成发布产物

知识存储和知识发布是两回事。

系统内部可以有：

- 候选知识
- 冲突知识
- 过时知识
- 待审知识
- 历史版本

但对外可见的页面，应该只来自治理后可发布的版本。

这一层可以产出：

- Wiki 主题页
- FAQ 页面
- SOP
- 故障排查页
- Playbook
- Agent 可读知识包

也就是说，发布页不是知识主存储，而是知识编译产物。

### 2.6 Consume：人类和 Agent 是两类不同消费者

一个容易被忽略的事实是：页面适合人读，不一定适合模型吃。

所以同一份知识底座，通常需要两种出口：

- `human-readable page`
- `agent-readable pack`

前者强调结构清晰、层次自然、引用可展开；后者强调 compact、可筛选、可带 scope、可按置信度裁剪。

如果不把这两类读取面拆开，最后往往会两边都不满意。

## 3. 一个可落地的 LLM Wiki，架构上至少要补哪几层

如果从模块角度去拆，一个更稳的 `LLM Wiki` 通常至少要有下面这 8 个模块：

1. `Connector Service`
2. `Normalizer`
3. `Evidence Store`
4. `Knowledge Extractor`
5. `Governance Engine`
6. `Knowledge Store`
7. `Wiki Compiler`
8. `Runtime API`

可以把它们归到 5 层里：

### 3.1 Source Layer

负责接入外部和内部知识源。

### 3.2 Evidence Layer

负责保存规范化证据，解决可追溯问题。

### 3.3 Knowledge Layer

负责知识抽取、融合、冲突处理和版本化。

### 3.4 Publish Layer

负责把知识编译成页面、FAQ、SOP、Playbook。

### 3.5 Runtime Layer

负责对用户和 Agent 暴露查询、反馈、写回和重编译接口。

这套拆法的价值在于：每一层只解决一类问题，不会把“采集、抽取、治理、发布”混成一团。

## 4. 为什么“知识单元”是这类系统的分水岭

如果说 RAG 系统和 `Knowledge OS` 最大的区别是什么，我会说是：**有没有知识单元这一层。**

### 4.1 Chunk 适合召回，不适合治理

chunk 的优点是简单、快、天然适合检索。

但 chunk 很难回答这些问题：

- 这是不是同一条知识的不同表述
- 这条知识是稳定事实还是临时经验
- 它适用于哪个产品版本和环境
- 新证据到来时是“增强支持”还是“引入冲突”

chunk 是检索单位，不是知识治理单位。

### 4.2 知识单元的核心，不是正文，而是身份

一个真正能落地的知识单元，不能只靠标题和摘要来区分。

更稳的方式是给它一个稳定 identity，例如：

`claim identity = subject + assertion + applicability_scope + valid_time_range`

也就是说，你要先能回答：

- 这条知识在说谁
- 它到底在断言什么
- 它适用于哪里
- 它在什么时间内有效

只有这样，系统后面才能做：

- 去重
- 冲突检测
- 版本推进
- 多 scope 并存
- 幂等处理

### 4.3 证据集合变化，不等于知识身份变化

这是实践里特别容易踩坑的一点。

同一条知识如果多了一条支持证据，它不应该被系统误判成“新的知识”。

更合理的设计是：

- `canonical_key` 决定知识身份
- `support_revision` 决定支撑证据版本

这样系统才能避免：

- 重复单元膨胀
- 页面频繁抖动
- 回放任务反复制造新分支

## 5. 自动写回为什么最难，难点不在模型而在治理

很多团队做 `LLM Wiki` 时最兴奋的一件事，是让 Agent 自动生成和更新 Wiki。

但真正上线时，最危险的也是这一步。

因为“会生成页面”和“能持续写回而不失控”是两件完全不同的事。

### 5.1 没有证据绑定，写回一定会失真

如果一条知识不能追溯到来源，系统就没法判断：

- 它从哪来
- 是不是还有效
- 被谁纠正过
- 出现问题时该不该撤回

所以，一个成熟系统里，不应该有“来源不明的正式知识”。

### 5.2 没有冲突治理，写回会把知识库越写越乱

新知识和旧知识冲突时，系统不能只会“覆盖旧答案”。

更合理的路径是：

- 进入 `conflicted`
- 生成新版本
- 按 scope 并存
- 或进入审查队列

如果没有这层，系统会很快从“持续学习”变成“持续污染”。

### 5.3 没有时效性，旧经验会误导新任务

运行经验、临时 workaround、故障排查技巧，通常天然带时间属性。

它们今天成立，不代表三个月后还成立。

所以这类知识必须有：

- freshness score
- 衰减规则
- 过期下架
- 重新验证

否则最会害人的，往往不是“错误知识”，而是“过时但看起来很对的知识”。

### 5.4 没有发布门槛，候选内容会直接污染正式层

一个稳的系统一定要区分：

- 候选知识
- 已发布知识
- 冲突知识
- 已归档知识
- 已撤回知识

系统内部可以存很多东西，但正式发布层必须克制。

换句话说，**写回是 append 容易，publish 很难。**

## 6. 权限传播为什么是很多 LLM Wiki 最容易忽略的坑

一旦知识源开始混合，权限问题会比检索问题更快爆出来。

比如：

- 某个 FAQ 页面引用了来自高权限工单的结论
- 某个专题页混入了只对部分租户可见的操作经验
- Agent 在低权限上下文里读到了高权限证据的摘要

如果 ACL 只停留在原始来源层，系统几乎一定会在派生产物里出问题。

更稳的做法是：

1. `Evidence` 继承来源 ACL
2. `KnowledgeUnit` 取支撑证据 ACL 的交集
3. `WikiPage` 不做单页求交，而是按 ACL cohort 生成多个 page variant
4. `agent-readable pack` 只能返回调用方可见内容

核心原则是：

`deny-dominates`

也就是宁可少发，也不要因为摘要、聚合或运行时裁剪而泄露不该看到的信息。

## 7. 一个更靠谱的第一版 MVP 应该收敛到哪里

这类系统最容易犯的错，就是一开始就想做“万能知识中台”。

更实际的做法是：先把系统收敛到一个能闭环的小范围。

### 7.1 输入范围先收敛

第一版建议只接：

- Markdown 文档
- Agent 执行日志
- 工单 / 问题单
- 聊天纪要 / 复盘记录

### 7.2 知识类型先收敛

第一版建议只做：

- `fact`
- `procedure`
- `failure_pattern`
- `faq`

### 7.3 输出产物先收敛

第一版建议只发：

- 主题 Wiki
- FAQ 页面
- 故障排查页

### 7.4 自动化边界也要收敛

第一版可以自动做：

- 新增知识单元
- 补充证据
- 更新 confidence / freshness
- 标记过时
- 触发页面重编译

第一版不建议自动做：

- 重写高权威主文档
- 删除正式页面
- 跨专题合并整页
- 修改权限 / 制度类文档
- 覆盖存在冲突的正式结论

这不是保守，而是在给系统时间建立治理能力。

## 8. 技术选型上，为什么 v1 不建议一开始就上“大图谱”

很多人一听“知识中台”“关系”“版本”“冲突”，会直觉想到知识图谱和图数据库。

它们当然有价值，但第一版不一定需要这么重。

如果目标是尽快把 `Evidence -> Unit -> Publish` 路径跑通，一个更实际的组合通常就够了：

- `Postgres`
- `pgvector`
- `Postgres FTS`
- `S3 / MinIO`
- `Temporal / Inngest / 队列系统`

模型分工也不需要太复杂，通常三类就够：

- 抽取模型
- Judge / 评分模型
- 页面编译模型

对第一版来说，最重要的不是“技术栈听起来多高级”，而是：

- 能不能支持知识身份
- 能不能支持版本与撤回
- 能不能支持 ACL 传播
- 能不能支持发布与回滚

这些控制面做好了，系统才有资格扩成更大的知识图谱平台。

## 9. 从工程视角看，LLM Wiki 最终要解决的到底是什么

如果把前面的内容压缩一下，我会认为一个真正可落地的 `LLM Wiki` 至少要解决 5 件事：

### 9.1 不只是“找到内容”，而是“找到可发布知识”

检索是入口，不是终点。

### 9.2 不只是“保存内容”，而是“沉淀可追溯证据”

没有证据层，后面所有治理都站不住。

### 9.3 不只是“生成页面”，而是“管理知识生命周期”

知识会新增、冲突、过时、撤回、重编译，这些都需要系统原生支持。

### 9.4 不只是“让 Agent 写”，而是“让系统约束 Agent 写”

自动写回真正重要的不是生成速度，而是边界、门槛和回滚。

### 9.5 不只是“做一个聊天入口”，而是“做一个知识操作系统”

最终价值不是多一个 AI UI，而是让组织里的知识第一次具备：

- 持续沉淀能力
- 结构化治理能力
- 版本与撤回能力
- 可发布和可复用能力

---

## 10. 一句话总结

`LLM Wiki` 不应该被实现成“AI 自动改文档”，而应该被实现成“Agent 持续产生知识事件，系统对事件做治理，再编译成可发布的 Wiki 与可消费的知识包”。

如果只把它理解成一个更智能的 RAG 页面，你大概率只能得到一个更会回答问题的文档站。

但如果把它理解成一套知识加工、治理和发布系统，你才有机会真正把它做成一个可持续演化的 `Knowledge OS`。
