# AI 可以直接跑 `rm -rf` 吗？生产级权限系统的四层防线设计

## 1. 问题定义

当 AI Agent 从“回答问题”走向“执行动作”时，一个绕不过去的问题是：

> 我们是否应该允许 AI 直接获得类似 `rm -rf` 这样的高危执行能力？

从实验环境看，答案似乎可以是“可以，只要模型足够聪明”；  
但从生产系统视角看，这个问题的关键从来不是“敢不敢”，而是：

> **如何让 AI 即使判断错误、工具误用、策略漏配，依然无法把生产环境直接带进灾难。**

因此，真正需要设计的不是“AI 能不能执行危险命令”，而是：

- AI 看到的到底是“命令”，还是“受限能力”
- 谁来判定一个动作是否允许执行
- 高风险动作是否必须经过审批
- 真出错时，系统是否可追踪、可熔断、可恢复

这也是生产级权限系统必须建立“四层防线”的原因。

---

## 2. 为什么不能把高权限 Shell 直接交给 AI

很多系统在早期做 Agent 落地时，会默认把“执行能力”理解为：

- 能调用 shell
- 能访问服务器
- 能修改文件
- 能操作数据库
- 能执行部署脚本

这种设计在 demo 阶段很直接，但在真实生产环境里风险极高。因为 `rm -rf` 的危险性不在于这四个字符本身，而在于它代表了一种更底层的问题：

> **系统把“任意执行”暴露给了一个基于概率生成决策的主体。**

一旦这样设计，风险通常来自以下几个方面。

### 2.1 模型理解错误

AI 可能误解用户目标、误判路径范围、误读环境上下文。  
例如本来要清理临时目录，却误删了持久化目录。

### 2.2 工具边界不清

如果系统只给了一个“执行任意命令”的入口，那么权限控制就退化成了“希望模型自己小心”。  
这不是权限设计，而是把安全责任外包给模型行为。

### 2.3 环境耦合过深

一旦 AI 直接连上生产主机、数据库或云控制面，任何错误都可能直接演化成生产事故，而不是一次可控失败。

### 2.4 缺少可恢复机制

如果危险动作本身没有版本化、快照、软删除或审计记录，那么误删一旦发生，损失可能不可逆。

所以问题的本质不是：

> “AI 会不会犯错？”

而是：

> **系统是否允许 AI 的一次错误直接穿透到底层资源。**

---

## 3. 设计目标：把“命令执行”改造成“受控能力调用”

一个生产级 AI 权限系统，不应该把 AI 当成拥有自由操作权的管理员，而应把它设计成一个受约束的执行主体。

它的目标不是“让 AI 更自由”，而是同时满足四件事：

1. AI 不直接拥有无限命令执行权
2. 所有危险动作都能被结构化判定，而不是靠自然语言临场理解
3. 高风险操作默认需要额外审批，而不是默认可执行
4. 即使最终执行出错，也必须可审计、可熔断、可恢复

如果用一句话概括：

> **不给 AI 裸权限，只给 AI 在明确边界内调用受控能力的资格。**

---

## 4. 总体架构：五个关键组件

要实现这一点，系统通常需要五个核心组件协同工作。

### 4.1 `Agent Runtime`

负责理解任务、生成计划、选择工具。  
它可以“提出想做什么”，但不能直接拿着高权限去触碰生产资源。

### 4.2 `Capability Gateway`

这是 AI 的统一执行入口。  
所有外部动作都通过它暴露成受控能力，而不是让 AI 自己拼命令。

例如，不暴露：

```text
rm -rf /data/xxx
```

而是暴露：

```text
delete_temp_files(namespace, older_than, limit, dry_run)
```

这意味着 AI 面对的是结构化动作，而不是无限 shell。

### 4.3 `Policy Engine`

负责实时授权判定。  
它根据主体、动作、资源、环境、条件和风险等级，决定当前请求是：

- 允许
- 拒绝
- 进入审批
- 降级为 dry-run
- 强制进入只读模式

### 4.4 `Approval Service`

所有高风险操作统一在这里走审批。  
AI 可以产出执行计划，但不能单方面完成高风险落地。

### 4.5 `Audit & Recovery Plane`

负责全链路审计、异常检测、限流、熔断和回滚。  
这是系统从“能跑”走向“生产级”的关键补强层。

## 5. 四层防线

四层防线的目标不是重复授权，而是在不同阶段阻断不同类型的风险。

### 5.1 第一层：策略防线

这一层解决的问题是：

> **AI 在原则上被允许做什么，不被允许做什么。**

核心原则有四条：

- 默认拒绝，而不是默认允许
- 授权对象是“能力”，不是“通用 shell”
- 授权必须绑定资源和条件
- 高风险动作必须显式分类，不能模糊处理

也就是说，系统不应该定义这种粗粒度权限：

- `ops-admin`
- `server-maintainer`
- `prod-shell-access`

而应定义成明确能力，例如：

- 可读某服务日志
- 可重启某环境服务
- 可删除某命名空间下过期缓存
- 不可删除永久数据目录
- 不可修改权限策略本身
- 不可访问 secrets 存储

这一层的意义在于：

> **让 AI 即使“想执行危险事”，在制度层面也根本拿不到那种能力。**

### 5.2 第二层：执行防线

策略允许还不够，执行环境本身也必须是受限的。  
这一层解决的问题是：

> **就算策略漏了一条，AI 是否仍然做不出灾难性动作。**

这一层通常包含四类手段。

#### 5.2.1 不开放裸命令，只开放受控工具

不要把 `rm -rf` 交给 AI。  
应该把“删除”封装成受控能力，并在工具内部做参数验证、路径白名单、数量限制、软删除或预检查。

#### 5.2.2 执行身份最小化

AI 使用独立的 service account，而不是共享人类管理员权限。  
它不应具备：

- 交互式登录能力
- 跨环境提权能力
- 直接读 secrets 的能力
- 修改 IAM/ACL 的能力
- 修改权限系统本身的能力

#### 5.2.3 运行环境隔离

AI 动作最好运行在容器或沙箱内，而不是直接进入生产主机。  
常见隔离措施包括：

- 只挂载白名单目录
- 限制网络出站
- 禁止访问宿主机关键路径
- 禁止访问 metadata service
- 禁止符号链接越界
- 禁止跨 namespace 访问

#### 5.2.4 危险资源硬保护

某些资源不应该依赖策略判断，而应直接内建拒绝：

- 根目录
- 持久化数据卷
- 数据库文件目录
- 配置控制平面
- 凭证存储路径
- 权限系统配置

这一层的核心不是“让模型更谨慎”，而是：

> **就算模型错了，执行器也没有把事情做坏的自由度。**

### 5.3 第三层：审批防线

即使前两层都做了，生产系统也不能假设规则永远正确。  
因此高风险操作必须有一层显式的人机协同审批。

这一层解决的问题是：

> **当动作的破坏性、影响范围或不可逆程度超过阈值时，是否还允许 AI 单独完成执行。**

通常需要进入审批的动作包括：

- 删除类操作
- 批量写入或批量修改
- 生产环境配置变更
- 流量切换
- 数据迁移
- 权限策略更新
- 跨环境写操作
- 任何不可快速回滚的动作

审批系统应满足几个要求。

#### 5.3.1 AI 提交的是执行计划，不是直接动作

AI 在进入审批前，应先生成结构化计划，至少包括：

- 要做什么
- 为什么要做
- 会影响哪些资源
- 预计影响数量
- 风险等级
- 是否有 dry-run 结果
- 如何回滚

#### 5.3.2 审批对象必须可读、可判断

审批人看到的不能只是一句自然语言描述，而应看到：

- 目标资源
- 影响范围
- 差异摘要
- 风险标签
- 匹配到的策略规则
- 回滚方案
- 有效执行窗口

#### 5.3.3 风险分层审批

可以按风险等级设计：

- `P1`：自动允许
- `P2`：单人审批
- `P3`：双人审批
- `P4`：双人审批 + dry-run 强制 + 变更窗口
- `P5`：默认拒绝，仅 break-glass 特殊流程可放行

#### 5.3.4 审批结果必须一次一用

审批通过后，不应给 AI 一个更高角色，而应签发一次性、范围受限、短时有效的 execution token。

这一层的意义是：

> **把“模型决定是否执行”升级为“组织对高风险变更负责”。**

### 5.4 第四层：审计与恢复防线

前三层负责降低出错概率，第四层负责控制出错后果。  
这是很多系统最容易忽略、却最能体现生产级成熟度的一层。

这一层解决的问题是：

> **如果事故还是发生了，系统能否第一时间发现、停止、定位并恢复。**

主要包括五个方面。

#### 5.4.1 完整审计

系统应记录：

- 谁发起了任务
- 哪个 Agent / 模型版本参与
- 使用了哪些工具
- 命中了哪些策略
- 是否触发审批
- 审批人是谁
- 实际影响了哪些资源
- 执行结果如何

如果是 AI 系统，还应额外记录：

- 原始目标描述
- 计划摘要
- 工具参数快照
- policy version
- prompt / system rule version 的引用

#### 5.4.2 异常检测

不仅要看报错，还要看行为异常，例如：

- 删除量异常升高
- 高频调用危险工具
- 跨多个资源域快速变更
- 与历史模式显著不一致
- 审批后短时间内连续大规模执行

#### 5.4.3 配额与限流

所有 AI 执行动作都应有规模边界，例如：

- 单次最多操作多少对象
- 单任务最多影响多少资源
- 单分钟最多执行多少危险请求
- 某类工具的并发上限

#### 5.4.4 熔断与 Kill Switch

系统必须能在发现异常时快速切断：

- 某个 Agent 的写权限
- 某类工具的调用
- 某个环境的 AI 执行面
- 全局降级到只读模式

#### 5.4.5 可恢复设计

如果系统允许任何删除或修改动作，就必须事先设计恢复路径，例如：

- 软删除
- 延迟删除
- 回收站
- 对象版本化
- 数据快照
- WAL / binlog
- 变更前自动备份
- 配置版本回滚

这一层的核心理念是：

> **生产级系统不依赖“绝不出错”，而依赖“出错也不会失控”。**

## 6. 权限模型：从角色授权转向能力授权

传统 RBAC 很适合人类组织管理，但对 AI Agent 不够细。  
更适合 AI 系统的模型通常是：

```text
Principal + Capability + Resource + Environment + Condition + Risk
```

### 6.1 `Principal`

不只是谁“登录了系统”，还要表示是谁在以什么身份发起动作：

- `human_user`
- `ai_agent`
- `automation_workflow`
- `approval_delegate`
- `break_glass_operator`

对于 `ai_agent`，还建议额外记录：

- `agent_id`
- `agent_class`
- `on_behalf_of`
- `trust_tier`
- `tenant` / `team`
- `tool_scope`

### 6.2 `Capability`

权限对象不再是宽泛角色，而是原子能力：

- `logs.read`
- `service.restart`
- `cache.delete`
- `deployment.rollout`
- `db.query.readonly`
- `config.update`

能力必须满足三点：

- 语义单一
- 边界清晰
- 可单独审计和限流

### 6.3 `Resource`

资源必须结构化，不能只是一段自然语言路径。  
例如：

- 某个 service
- 某个 namespace
- 某个文件目录
- 某个数据库 schema
- 某个对象集合

### 6.4 `Condition`

条件用于表达细粒度限制，例如：

- 仅限 `staging`
- 仅限工作时间
- 仅限只读
- 仅限 `tmp/*`
- 单次删除上限 `500`
- 仅允许删除 `24` 小时前对象
- 必须先 dry-run
- 必须有 owner 审批

### 6.5 `Risk`

风险不应只是附属标签，而应进入授权流程。  
因为同一个 capability 在不同对象、不同规模、不同环境下，风险完全不同。

## 7. `AI Agent + Tool/MCP` 场景下的权限边界

如果系统使用 MCP 或类似工具调用框架，需要额外防止一类问题：

> **AI 通过工具链绕过权限边界。**

因此，建议把工具权限拆成三层。

### 7.1 工具可见性

不是所有工具都应该在当前轮对 AI 可见。  
例如规划阶段只给读工具，执行阶段才可能暴露有限写工具。

### 7.2 工具调用权限

即使工具可见，也不表示可以调用。  
每次调用都仍要经过策略引擎判定。

### 7.3 工具内部护栏

工具实现本身还应做二次保护，例如：

- 参数校验
- 路径规范化
- 越界检查
- 数量上限
- 幂等控制
- 操作前摘要确认
- 自动创建回滚点

也就是说：

> **权限判断不能只做在 Agent 层，还必须做在真正的执行闸门层。**

## 8. 危险操作审批流示意

如果把“删除类能力”作为一个典型高风险动作，推荐的审批流如下：

```text
用户 / 系统提出任务
  -> Agent 生成结构化执行计划
  -> Capability Gateway 做预检
  -> Policy Engine 计算风险
  -> 低风险：直接执行
  -> 中高风险：进入审批
  -> 审批通过后签发一次性 execution token
  -> 执行器校验 token、范围和时效
  -> 落地执行
  -> Audit & Recovery Plane 记录与监控
```

可以进一步将审批对象抽象为：

```ts
type ApprovalRequest = {
  requestId: string
  principal: {
    type: "ai_agent"
    agentId: string
    onBehalfOf?: string
    trustTier: "low" | "medium" | "high"
  }
  action: {
    capability: string
    operation: string
  }
  target: {
    resourceType: string
    environment: "dev" | "staging" | "prod"
    identifiers: string[]
  }
  impact: {
    estimatedObjectCount?: number
    blastRadius: "small" | "medium" | "large"
    reversibility: "easy" | "partial" | "hard"
  }
  safeguards: {
    dryRunCompleted: boolean
    backupCreated: boolean
    rollbackPlan: string
  }
  policy: {
    matchedRules: string[]
    riskLevel: "P1" | "P2" | "P3" | "P4" | "P5"
    decision: "allow" | "deny" | "approval_required"
  }
  summary: string
  expiresAt: string
}
```

审批通过后，不应该给 Agent 一个更高角色，而应签发一个范围收缩后的执行令牌，只允许：

- 本次 `request_id`
- 指定 `capability`
- 指定资源集合
- 指定参数摘要
- 指定有效时间窗口

这样才能防止“一次审批，多次复用”的权限扩散。

## 9. 对 `rm -rf` 的最终工程结论

如果回到最初的问题，一个生产级系统的答案应该是明确的。

### 9.1 不应允许的情况

以下条件下，不应让 AI 直接执行 `rm -rf`：

- 持有通用 shell
- 直接连生产主机
- 使用共享管理员账号
- 没有路径白名单
- 没有 dry-run
- 没有审批
- 没有审计
- 没有恢复机制

### 9.2 可以接受的情况

只有在以下条件同时成立时，系统才可以允许 AI 使用“删除类能力”：

- 删除能力已被工具化封装
- 资源范围受白名单约束
- 路径已规范化且防越界
- 单次影响规模有限
- 高风险场景已审批
- 执行身份是最小权限 service account
- 全程可审计
- 出错后可恢复

此时允许的也不是 `rm -rf` 本身，而只是：

> **一个被强约束、可追踪、可回滚的删除能力。**

---

## 10. 总结

“敢不敢让 AI 跑 `rm -rf`”这个问题，表面上是在问 AI 是否足够可靠，  
但在工程上，它真正考验的是：

> 系统是否把 AI 的执行能力设计成了一个可治理的权限问题，而不是一个对模型行为的信任问题。

生产级权限系统的关键，不是期待 AI 永远不犯错，而是通过四层防线把错误限制在可控范围内：

- 第一层防策略过宽
- 第二层防执行失控
- 第三层防高风险误判
- 第四层防事故不可收拾

最终目标不是“让 AI 拿到高权限”，而是：

> **让 AI 只能在受控能力、明确边界和可恢复机制内完成工作。**
