# 死循环、重复犯错、Token 烧穿：你的 Agent Loop 缺这三个“保险丝”

## 一句话总结

很多 Agent 系统的问题，不是“模型不会”，而是 `loop control` 没设计好。一个能上线的 Agent Loop，至少要有三道保险丝：

- **终止保险丝**：防止死循环
- **纠错保险丝**：防止重复犯错
- **预算保险丝**：防止 Token 烧穿

如果把 Agent 只当作“能连续行动的模型”，系统通常会越跑越失控；如果把 Agent 看作一个需要被约束的决策回路，这三层控制就会变成基础设施，而不是补丁。

---

## 1. 问题定义

很多团队在做 Agent 时，最先怀疑的往往是模型能力不够。

但真实进入工程环境后，更常见的故障模式其实是：

- Agent 一直在调用工具，却没有真正推进任务
- Agent 在同一类错误上反复踩坑，像没有学到任何经验
- Agent 每一轮都把更多历史、检索结果、工具输出塞进上下文，导致成本和延迟一路上涨

这三类现象表面不同，本质上都说明一件事：

> **系统只设计了“怎么继续行动”，却没有设计“什么时候该停、怎么避免重复出错、如何控制成本”。**

也就是说，问题不在于 Agent 会不会执行，而在于它有没有被安全地约束执行。

如果压缩成一句工程表达：

`Reliable Agent Loop = progress control + failure learning + token budgeting`

---

## 2. 为什么 Agent Loop 容易失控

一个最朴素的 Agent Loop 往往长这样：

1. 读取用户目标
2. 思考下一步
3. 调用工具
4. 读取结果
5. 再思考下一步
6. 直到模型自己决定结束

demo 阶段，这样的流程通常已经能跑通。

但一旦任务拉长、工具变多、上下文变复杂，系统就会暴露出 3 个结构性缺陷：

### 2.1 没有显式退出条件

只要模型还能生成下一步，loop 就会继续跑。

于是系统会把“还在做事”误判成“还在推进”。  
这正是死循环出现的土壤。

### 2.2 没有显式失败沉淀

很多系统虽然保留了日志，却没有把失败提炼成后续决策约束。

于是 Agent 能看到“刚刚发生了什么”，却无法真正记住“刚刚为什么失败、下次不要怎么做”。

### 2.3 没有显式预算治理

上下文如果只会追加，不会选择、排序、压缩和重置，就会迅速出现：

- 输入 token 越来越高
- 响应延迟越来越长
- 模型注意力越来越分散
- 最重要的信息被噪音稀释

所以，Agent Loop 的核心难题从来不只是“能不能连起来跑”，而是：

> **能不能在长时、多轮、多工具的环境下持续收敛，而不是持续膨胀。**

---

## 3. 第一层保险丝：终止保险丝，解决死循环

死循环并不一定表现为完全相同的动作反复执行。

更常见的是一种“伪忙碌”状态：

- 不断重复搜索相似资料
- 同一个命令失败后继续重试
- 在“再检查一下”和“再试一次”之间来回切换
- 明明任务已经没有推进空间，却迟迟不升级、不结束

这类问题的根源通常不是模型完全不懂，而是系统没有回答下面几个问题：

- 什么叫完成
- 什么叫失败
- 什么叫没有进展
- 在什么条件下必须停止

### 3.1 终止保险丝应该控制什么

一个可上线的 Agent Loop，至少应该有下面 4 类终止控制。

#### 步数上限

给任务、阶段、工具调用设置最大步数。

一旦超过阈值，系统就不能继续默认“再想一步”，而是应该：

- 结束任务
- 进入降级路径
- 请求外部输入
- 或升级给人工

#### 重复动作检测

如果 Agent 连续多次执行高度相似的操作，比如：

- 调同一个工具
- 读取同一批资源
- 对同一类错误做近似修复

系统就应该判定为循环风险，而不是让模型继续自由发挥。

#### 进展判定

终止控制不能只看“做了多少步”，还要看状态有没有实质变化，例如：

- 是否拿到了新的证据
- 是否缩小了问题范围
- 是否消除了一个阻塞条件
- 是否完成了某个子目标

没有状态变化的多步执行，本质上就是空转。

#### 升级路径

当系统判定“无法继续推进”时，不能只会失败退出。

更合理的设计是准备几条显式路径：

- 切换策略
- 缩小目标
- 请求补充信息
- 交接给新的 agent 或人工流程

### 3.2 终止保险丝的本质

终止保险丝不是让 Agent 少做事，而是防止系统把“连续动作”误当成“有效进展”。

如果没有这一层，Agent 的默认行为通常会退化成：

> **只要还能生成下一步，就继续跑。**

这不是自治，而是失控。

## 4. 第二层保险丝：纠错保险丝，解决重复犯错

比死循环更隐蔽的问题，是重复犯错。

很多 Agent 不会卡在完全相同的一步，但会反复掉进同一种坑里：

- 某个命令已经因为权限问题失败过，后面又重新执行
- 某条修复路径已经验证无效，仍然继续沿着它微调
- 某个工具在当前场景下返回了不可靠结果，系统却没有形成稳定判断

这类问题说明系统里有“历史”，但没有“经验”。

### 4.1 日志不等于学习

很多系统误以为：只要把完整执行轨迹保留下来，Agent 就自然会从中学习。

但原始日志通常只适合回放，不适合约束下一步决策。

真正需要沉淀的是结构化失败信息，例如：

- 尝试了什么
- 为什么失败
- 失败是偶发还是结构性
- 下次遇到类似情况该避免什么

换句话说，**失败必须从“发生过的事件”变成“后续策略的约束”**。

### 4.2 纠错保险丝应该包含什么

#### 失败指纹化

不要只保存原始报错文本，而要提炼出失败模式。

例如，系统不该只记住：

`422 Unprocessable Entity`

更有价值的是记住：

`路由匹配顺序错误，导致 reorder 被解析为 frame_id`

后者才足够约束后续推理。

#### 已尝试方案列表

每个长任务都应该维护一个结构化 scratchpad，用来记录：

- 已尝试方案
- 观察结果
- 失败原因
- 下一步避免重复尝试的约束

这样 Agent 的后续搜索不是从零开始，而是在被压缩过的经验空间中继续。

#### 重试升级策略

重试不应该等于“原样再来一次”。

更合理的是做分层升级：

1. 第一次失败：允许同路径小范围重试
2. 第二次失败：换参数或缩小范围
3. 第三次失败：换工具或换策略
4. 多次失败：强制总结并升级给外部流程

如果系统没有这类升级阶梯，所谓“重试机制”最终只会变成“重复犯错机制”。

### 4.3 纠错保险丝的本质

一个可靠的 Agent 不是不会出错，而是：

> **同一个错误模式，不会以原样一遍又一遍出现。**

如果系统只能累积日志，不能累积约束，Agent 看起来跑了很多轮，实际上每一轮都像第一次出错。

## 5. 第三层保险丝：预算保险丝，解决 Token 烧穿

最贵也最常见的问题，是 Token 烧穿。

很多 Agent 系统一开始表现不错，但随着任务轮次增加，会越来越像一个不断背包加重的执行器：

- 聊天历史一直保留
- 工具结果整段回填
- 检索结果全量拼接
- memory、profile、policy 一路叠加

最后系统把“记得更多”误当成“更聪明”，却忽略了长上下文的真实代价。

### 5.1 Token 烧穿的后果不只是贵

Token 问题通常会带来三重损失：

#### 成本上升

每一轮都在重复支付大量旧上下文的输入成本。

#### 延迟上升

上下文越长，prefill 越重，交互体验越差。

#### 质量下降

这是最容易被低估的一点。

上下文变长后，问题不只是“更贵”，还包括：

- 重要约束被淹没
- 模型注意力分散
- 工具日志污染决策上下文
- 旧信息和新任务阶段不再匹配

也就是说，`more context` 并不天然等于 `better context`。

### 5.2 预算保险丝应该怎么设计

一个成熟的 Agent Loop，不应该把上下文当作追加日志，而应该当作受限预算。

至少要有 4 类能力。

#### Selection：这一步到底需要什么

系统必须决定本轮到底要不要带上：

- recent history
- memory
- retrieval results
- tool outputs
- workflow state
- policy blocks

不是所有来源每一轮都必须参与。

#### Ranking：谁更重要

不同上下文的重要性并不相同。

通常来说：

- system policy 高于旧聊天
- 当前任务状态高于长工具日志
- 高相关知识片段高于原始检索堆叠

如果不显式表达优先级，系统最终就会退化成：

> 谁先拼进去，谁活下来。

#### Budgeting：预算怎么分

上下文窗口不是拿来“尽量塞满”的，而是拿来分配资源的。

一个典型做法是按来源预留槽位，例如：

- instructions / policy
- current task state
- recent critical history
- retrieval snippets
- tool outputs
- reserved output space

这样能避免某一个来源把整个窗口挤爆。

#### Compression / Reset：超预算怎么办

超预算时，系统不能只有“删后面的内容”这一种动作。

更合理的处理层级通常是：

1. 去重
2. 结构化提炼
3. 片段级裁剪
4. 摘要压缩
5. 延迟加载
6. 必要时做上下文重置，通过结构化交接继续任务

### 5.3 预算保险丝的本质

预算保险丝不是要求系统“少给上下文”，而是要求它：

> **只给这一步真正有价值的上下文。**

否则，Agent Loop 很容易从“多轮执行系统”退化成“多轮堆料系统”。

---

## 6. 三个“保险丝”其实对应 Agent Loop 的三个基本问题

如果把 Agent Loop 抽象成一个持续决策系统，每一轮其实都在回答三个问题：

1. **还要不要继续？**
2. **刚才的失败学到了什么？**
3. **下一轮应该花多少上下文成本？**

这三个问题分别对应：

- 终止保险丝
- 纠错保险丝
- 预算保险丝

也就是说，这三道保险丝不是外围增强项，而是 Agent Loop 的控制平面。

很多系统之所以在 demo 阶段显得很聪明、到真实任务里却越来越不稳定，就是因为它们只设计了：

- 调工具
- 看结果
- 继续思考
- 再调工具

却没有设计：

- 何时停止
- 如何记错
- 怎样控本

于是默认行为就变成：

> **只要还能跑，就继续跑。**

这不是长时自治，而是长时失控。

---

## 7. 工程上怎么落地这三层控制

如果你正在设计一个真实可上线的 Agent 系统，可以把这三层保险丝分别落到下面几个模块里。

### 7.1 Loop Controller

负责：

- 步数上限
- 阶段切换
- 重复动作检测
- 无进展判定
- 终止 / 降级 / 升级

这一层决定 Agent 什么时候不该再继续跑。

### 7.2 Failure Memory / Scratchpad

负责：

- 保存已尝试方案
- 提炼失败模式
- 记录无效路径
- 输出下一步约束

这一层决定 Agent 会不会重复犯同一种错误。

### 7.3 Context Manager / Budget Controller

负责：

- 选择上下文来源
- 对上下文排序
- 分配 token 预算
- 做压缩、裁剪、延迟加载和重置

这一层决定 Agent 会不会把成本和上下文噪音一路堆高。

从系统设计角度看，三者分别回答的是：

- `Loop Controller`：能不能停
- `Failure Memory`：能不能学
- `Context Manager`：能不能控

---

## 8. 总结

很多 Agent 系统的真正问题，不是模型不够强，而是 loop 没有控制面。

当系统缺少终止保险丝时，Agent 会死循环；  
缺少纠错保险丝时，Agent 会重复犯错；  
缺少预算保险丝时，Agent 会把 token、延迟和上下文噪音一起烧穿。

所以，生产级 Agent 的核心能力不应该只定义为“能自主执行多久”，而应该定义为：

> **能否在长时、多轮、多工具环境里持续收敛。**

真正稳定的 Agent，不是最能跑的，而是最会在正确的时候停、在失败之后学、在每一步都知道该花多少上下文成本的那个。
