# 应用层大模型 Token 优化策略

## 1. Prompt 工程优化

### 精简 System Prompt

- 删除冗余描述，用结构化格式（JSON/YAML）替代自然语言长句
- 把不变的指令放 system prompt，利用 **Prompt Caching**（Anthropic 缓存命中成本降 90%）
- 定期审计 prompt 模板，移除过时或重复的指令段

### Few-shot 示例最小化

- 只保留最具代表性的 1-2 个示例，而非堆砌 5-6 个
- 用 Chain-of-Thought 引导替代大量示例演示

## 2. 上下文裁剪

### RAG 召回精准化

- 召回 top-k 结果后做 **rerank**，只送最相关的 2-3 条进上下文
- 对长文档做摘要后再注入 prompt，而非原文整段塞入
- 控制每条 chunk 的长度，避免无关段落混入

### 对话历史压缩

| 策略 | 做法 | 效果 |
|------|------|------|
| 滑动窗口 | 只保留最近 N 轮 | 简单粗暴，丢失早期上下文 |
| 摘要替换 | 每 N 轮用小模型生成摘要替换原文 | 平衡上下文保留与 token 节省 |
| 关键信息提取 | 只保留实体 / 决策 / 结论，丢弃寒暄 | 压缩率最高 |

```
原始 10 轮对话: ~4000 tokens
压缩为摘要:     ~300 tokens   （节省 ~90%）
```

## 3. 模型路由（Model Router）

用轻量分类器（规则引擎 / 小模型）做意图路由，80% 的请求可以走便宜模型：

```
用户请求
  │
  ├─ 简单（分类 / 提取 / 格式化）→ Haiku        ($0.25/M tokens)
  ├─ 中等（生成 / 分析 / 编码）  → Sonnet       ($3/M tokens)
  └─ 复杂（推理 / 架构 / 创作）  → Opus         ($15/M tokens)
```

**路由判断依据**：

- 输入长度、任务类型关键词
- 是否需要多步推理
- 用户等级 / 业务优先级

## 4. 缓存策略

| 层级 | 方法 | 效果 |
|------|------|------|
| **Prompt Cache** | Anthropic API 的 `cache_control` 标记静态前缀 | 缓存命中部分输入成本降 90% |
| **语义缓存** | 对请求做 embedding，相似度 > 0.95 直接返回缓存结果 | 完全避免重复调用 |
| **KV Cache 复用** | 共享前缀的多轮对话复用计算结果 | 减少推理侧重复计算 |

### Prompt Cache 使用示例

```json
{
  "system": [
    {
      "type": "text",
      "text": "大段不变的系统指令...",
      "cache_control": { "type": "ephemeral" }
    }
  ]
}
```

适用场景：system prompt 较长且多次请求共享同一前缀。

## 5. 输出控制

### 限制输出长度

- 设置合理的 **`max_tokens`**，避免模型长篇大论
- 在 prompt 中明确字数 / 条目约束："用 3 句话总结"

### 结构化输出

```python
# ✗ 不好：自由文本输出
"请详细分析这个问题..."           # 输出 ~2000 tokens

# ✓ 好：要求结构化返回
"以JSON返回: {result, confidence}" # 输出 ~50 tokens
```

利用 **Tool Use / Function Calling** 强制模型只返回必要字段。

### 流式中断

检测到答案已完整时提前 abort 流式连接，避免生成冗余 token。

## 6. 请求合并与批处理

### Batch API

非实时场景使用批处理接口，**成本直接减半**（Anthropic Batch API 50% 折扣）。

### 合并同类请求

```python
# ✗ 不好：3 次调用，system prompt 重复 3 次
for item in items:
    call_llm(system_prompt + item)

# ✓ 好：1 次调用，system prompt 只传 1 次
call_llm(system_prompt + "处理以下3项:\n" + "\n".join(items))
```

## 7. 分层处理（Cascade）

```
原始输入（长文档 / 大量数据）
  │
  ├─ 第一层: 小模型做预处理（提取、过滤、摘要）  → 成本低
  │
  └─ 第二层: 大模型基于精华内容做深度推理        → 输入量小，成本可控
```

典型场景：

- 长文档 QA：先用 Haiku 提取相关段落，再用 Sonnet 回答
- 代码审查：先用小模型做 lint 级检查，复杂问题才升级到大模型
- 数据分析：先用规则 / 小模型做数据清洗，再让大模型做洞察
