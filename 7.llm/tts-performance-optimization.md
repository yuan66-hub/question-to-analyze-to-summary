# 文本转语音（TTS）性能优化策略

## 1. 流式合成（Streaming Synthesis）

核心思路：不等整段文本合成完毕，边合成边播放。

```
传统方式: [全部文本] → [完整音频] → [播放]  （高延迟）
流式方式: [句子1] → [音频1] → [播放1]
          [句子2] → [音频2] → [播放2]  （低感知延迟）
```

关键技术：

- **文本分句**：按标点符号、语义边界切分，避免跨句合成
- **音频拼接**：处理片段之间的过渡平滑（crossfade），避免爆音
- **预取队列**：当前片段播放时，后续片段已在合成

## 2. 模型层优化

| 优化方向 | 方法 | 效果 |
|---------|------|------|
| **模型蒸馏** | 用大模型指导训练小模型 | 推理速度提升 3-5x |
| **量化** | FP32 → INT8/FP16 | 内存减半，速度提升 1.5-2x |
| **剪枝** | 去除冗余神经元/注意力头 | 减少 20-40% 计算量 |
| **算子融合** | 合并连续算子（如 LayerNorm + Linear） | 减少内存访问开销 |
| **KV Cache** | 自回归解码中缓存已计算的 Key/Value | 避免重复计算 |

## 3. 声码器（Vocoder）加速

声码器是 TTS 的性能瓶颈（将 Mel 频谱转为波形）：

- **HiFi-GAN** → 轻量变体（MB-MelGAN），速度提升 10x+
- **分组卷积**：减少卷积参数量
- **多频段合成**：将全频信号拆分为子频段并行合成，最后合并
- **ONNX/TensorRT 部署**：利用硬件加速推理

## 4. 缓存策略

```
┌─────────────────────────────────┐
│         缓存层级设计              │
├─────────────────────────────────┤
│ L1: 完整句子缓存（精确匹配）       │  ← 命中率高的常用语句
│ L2: 音素序列缓存                  │  ← 相同发音不同文字
│ L3: 说话人嵌入缓存                │  ← 多次请求同一音色
│ L4: 韵律模板缓存                  │  ← 相似句式复用韵律
└─────────────────────────────────┘
```

- **热门短语预合成**：高频语句提前离线合成
- **音素级缓存**：相同音素序列直接复用中间表征
- **LRU 淘汰**：基于访问频率管理缓存容量

## 5. 并行化与调度

```
请求 → 文本前端 → [句子1] ──→ 声学模型 ──→ 声码器 ──→
                  [句子2] ──→ 声学模型 ──→ 声码器 ──→ 音频拼接 → 输出
                  [句子3] ──→ 声学模型 ──→ 声码器 ──→
```

- **Pipeline 并行**：前端分析、声学建模、波形生成形成流水线
- **Batch 推理**：多个请求合并为一个 batch，提升 GPU 利用率
- **Dynamic Batching**：按序列长度分桶，减少 padding 浪费
- **异步 I/O**：音频写入和网络传输不阻塞推理线程

## 6. 文本前端优化

文本前端（Text Frontend）也是延迟来源：

- **多音字消歧**：用轻量 BERT 或规则引擎替代大模型
- **韵律预测**：简化韵律模型，或用规则 + 统计混合方案
- **数字/日期归一化**：预编译正则 + 查表，避免运行时解析开销
- **G2P 缓存**：字 → 音素映射建立词典缓存

## 7. 部署架构优化

```
客户端                        服务端
┌──────┐    WebSocket      ┌──────────────┐
│ 前端  │ ◄──(流式音频)──── │  TTS Gateway  │
└──────┘                   └──────┬───────┘
                                  │ gRPC
                    ┌─────────────┼─────────────┐
                    ▼             ▼              ▼
              [TTS Worker 1] [TTS Worker 2] [TTS Worker 3]
                    │
              ┌─────┴─────┐
              │ GPU 推理池  │  ← 模型常驻显存，避免加载开销
              └───────────┘
```

- **WebSocket 流式推送**：替代 HTTP 轮询，减少传输延迟
- **模型预热**：服务启动时加载模型到 GPU，首次请求无冷启动
- **弹性伸缩**：根据 QPS 自动扩缩 Worker 实例
- **边缘部署**：在 CDN 边缘节点部署轻量模型，降低网络延迟

## 8. 关键性能指标

| 指标 | 说明 | 优化目标 |
|------|------|---------|
| **RTF**（Real-Time Factor） | 合成时长 / 音频时长 | < 0.1（即 10x 实时） |
| **首包延迟** | 请求到第一个音频包的时间 | < 200ms |
| **MOS**（Mean Opinion Score） | 主观音质评分（1-5） | > 4.0 |
| **吞吐量** | 每秒处理请求数 | 根据业务需求 |

核心原则：延迟优化的本质是**让用户尽早听到声音**，而非让整体合成更快。流式 + 缓存 + 模型轻量化三管齐下，可覆盖大部分场景。
