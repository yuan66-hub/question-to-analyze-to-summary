## 1. 需求场景与问题定义

### 1.1 典型业务场景

| 场景 | 素材特点 | 成片要求 |
|------|---------|---------|
| 电商商品视频 | 多角度商品拍摄、模特试穿、使用演示 | 突出卖点，15-60s，竖屏，批量多条 |
| 品牌营销 | 品牌素材库、活动现场视频、代言人片段 | 情感共鸣，30-120s，多平台适配 |
| 房产/汽车展示 | 室内外拍摄、航拍、细节特写 | 空间感呈现，60-180s，配专业解说 |
| 教育课程 | 录屏、PPT、教师出镜、实操演示 | 知识点清晰，5-15min，章节分段 |
| 短视频二创 | 多条短视频混剪素材 | 节奏感强，15-60s，差异化多条 |

### 1.2 用户输入模型

用户通过一次提交完成所有配置：

```
┌─────────────────────────────────────────────┐
│              用户输入                         │
│                                             │
│  📁 视频素材（1~N 条原始视频）                │
│  📝 产品卖点 / 核心文案（文本）               │
│  ⏱  目标时长（如 30s / 60s / 自动）          │
│  🔢 生成条数（如 3 条 / 5 条）               │
│  🏷  业务场景（电商 / 营销 / 教育 / ...）     │
│  🎨 风格偏好（活力 / 沉稳 / 科技 / 自动）     │
│  🎵 配乐偏好（无 / 自动匹配 / 指定曲目）     │
│  📐 输出比例（9:16 / 16:9 / 1:1）           │
└─────────────────────────────────────────────┘
```

### 1.3 核心挑战

**一句话概括**：从 N 条原始素材中自动提取精华片段，结合业务卖点，生成 M 条风格差异化的成品视频。

具体挑战分解：

| 挑战 | 难点 | 影响 |
|------|------|------|
| 素材理解 | 视频是非结构化数据，需要精确识别"哪段能用、哪段是废片" | 垃圾进垃圾出 |
| 卖点融合 | 文字卖点如何和视觉画面产生语义关联 | 成片是否"说人话" |
| 时长精确控制 | LLM 生成的脚本时长必须与目标时长精确匹配 | 用户体验底线 |
| 多条差异化 | 同一批素材生成多条视频不能雷同 | 投放效果和素材利用率 |
| 音画同步 | 配音节奏、字幕时间、BGM 节拍需要精确对齐 | 专业感 |
| 批量效率 | 多条视频并行生成的资源调度和耗时控制 | 可用性 |


### 2.2 五大模块职责

| 模块 | 职责 | 输入 | 输出 |
|------|------|------|------|
| **素材分析引擎** | 视频预处理、场景分割、内容理解、质量评分 | 原始视频文件 | 结构化素材标签库 |
| **LLM 编导引擎** | 融合卖点与素材，生成结构化分镜脚本 | 素材标签 + 用户配置 | 分镜脚本 JSON × M 条 |

| **渲染合成引擎** | 视频拼接、特效叠加、音频混合、编码导出 | 编排时间轴 + 素材文件 | 成品视频文件 × M 条 |
| **批量调度器** | 任务编排、并行控制、进度追踪、失败重试 | 全局任务配置 | 任务状态与结果 |

## 3. 素材理解与分析引擎

### 3.1 视频预处理

素材进入系统后的第一步是标准化处理：

```
原始视频 → 转码统一格式 → 关键帧抽取 → 场景分割 → 音视频分离
```

**转码统一**：不同设备拍摄的素材编码格式各异（H.264/H.265/ProRes），统一转为 H.264 MP4 + AAC 音频，避免后续处理的兼容性问题。

**关键帧抽取策略**：

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 固定间隔抽帧 | 每 N 秒抽取 1 帧 | 长视频快速索引 |
| 场景变化检测 | 基于帧间差异（SSIM/直方图）在场景切换点抽帧 | 多镜头素材 |
| 运动检测 | 在画面运动剧烈处加密抽帧 | 运动/动作类素材 |

**场景分割**：基于视觉特征的自动分割，将一条长视频拆分为多个语义连贯的片段（Shot）。

```typescript
interface VideoShot {
  shotId: string;
  sourceVideoId: string;
  startTime: number;      // 起始时间（秒）
  endTime: number;        // 结束时间（秒）
  keyframes: string[];    // 关键帧图片 URL
  thumbnailUrl: string;   // 代表帧缩略图
}
```

### 3.2 多模态内容理解

对每个分割后的片段进行多维度理解：

```
                    ┌──────────────────┐
                    │   视频片段 Shot    │
                    └────────┬─────────┘
            ┌────────────────┼────────────────┐
            ▼                ▼                ▼
     ┌──────────┐    ┌──────────┐    ┌──────────┐
     │ 视觉分析  │    │ 音频分析  │    │ 文本分析  │
     │          │    │          │    │          │
     │ 物体检测  │    │ ASR 语音  │    │ OCR 文字  │
     │ 场景分类  │    │  转文字   │    │  识别    │
     │ 动作识别  │    │ BGM 检测  │    │ 字幕提取  │
     │ 人脸/表情 │    │ 音量曲线  │    │          │
     │ 美学评分  │    │ 语种识别  │    │          │
     └──────────┘    └──────────┘    └──────────┘
            │                │                │
            └────────────────┼────────────────┘
                             ▼
                    ┌──────────────────┐
                    │  融合标签 & 摘要   │
                    └──────────────────┘
```

**视觉分析**：

- **物体检测**：识别画面中的核心物体（商品、人物、场景元素），建立与卖点的关联
- **场景分类**：室内/室外、工厂/展厅/街拍，为脚本编排提供场景上下文
- **动作识别**：走秀、使用演示、开箱等动作类型，用于匹配脚本中的动态镜头需求
- **美学评分**：基于构图、光线、清晰度对每帧打分，过滤低质量画面

**音频分析**：

- **ASR 语音识别**：提取人声内容转为文字，用于与卖点文案的语义关联
- **BGM 检测**：识别是否有背景音乐及其风格，避免成片中出现多层 BGM 叠加
- **音量曲线**：标记静音段、高音段，辅助后续音频编排

### 3.3 素材标签化与索引

分析结果以结构化标签形式存储，供 LLM 和编排引擎消费：

```typescript
interface ShotAnalysis {
  shotId: string;
  // 内容标签
  objects: string[];           // ["商品A", "模特", "包装盒"]
  scene: string;               // "室内-展厅"
  actions: string[];           // ["展示", "旋转", "试穿"]
  // 质量维度
  visualScore: number;         // 0-1 美学评分
  clarityScore: number;        // 0-1 清晰度
  stabilityScore: number;      // 0-1 稳定性（抖动检测）
  // 音频维度
  hasVoice: boolean;
  voiceText?: string;          // ASR 识别结果
  hasBGM: boolean;
  bgmStyle?: string;           // "轻快" | "沉稳" | "电子" | ...
  // 可用性
  usableDuration: number;      // 有效可用时长（去除抖动/虚焦段）
  emotionTag: string;          // "积极" | "平静" | "紧张" | ...
  sellingPointRelevance: Record<string, number>;  // 与各卖点的相关度
}
```

**与卖点的语义关联**是关键步骤：将用户输入的产品卖点（如"防水"、"轻便"、"高颜值"）与每个片段做向量相似度匹配，生成 `sellingPointRelevance` 映射。这使得 LLM 在编排脚本时能精确知道"哪段素材能证明哪个卖点"。

### 3.4 精华片段提取

并非所有素材都可用。精华提取的核心逻辑：

```
综合评分 = α × visualScore + β × clarityScore + γ × stabilityScore + δ × sellingPointRelevance
```

其中权重根据业务场景动态调整：

| 场景 | 美学权重 α | 清晰度权重 β | 稳定性权重 γ | 卖点相关度权重 δ |
|------|-----------|-------------|-------------|----------------|
| 电商 | 0.2 | 0.2 | 0.1 | **0.5** |
| 品牌营销 | **0.4** | 0.2 | 0.2 | 0.2 |
| 教育 | 0.1 | **0.4** | 0.2 | 0.3 |

低于阈值的片段标记为 `unusable`，不参与后续编排。

## 4. LLM 编导引擎

编导引擎是系统的核心大脑，负责理解全局意图并生成结构化分镜脚本。

### 4.1 输入融合策略

LLM 的输入由四部分组成：

```
┌─────────────────────────────────────────────────────┐
│                  LLM Context                        │
│                                                     │
│  ┌─────────────┐  ┌─────────────┐                   │
│  │ System      │  │ 素材摘要     │                   │
│  │ Prompt      │  │ (标签化后的  │                   │
│  │ (角色定义 +  │  │  结构化数据) │                   │
│  │  输出格式)   │  │             │                   │
│  └─────────────┘  └─────────────┘                   │
│  ┌─────────────┐  ┌─────────────┐                   │
│  │ 业务输入     │  │ 约束条件     │                   │
│  │ (卖点文案 +  │  │ (时长/条数/  │                   │
│  │  场景标签)   │  │  比例/风格)  │                   │
│  └─────────────┘  └─────────────┘                   │
└─────────────────────────────────────────────────────┘
```

**素材摘要压缩**：原始分析数据量大（每条视频可能有数十个 Shot），直接塞入 LLM 会超出 token 限制。压缩策略：

1. 按质量评分排序，只传入 Top-K 个高质量片段
2. 合并语义相似的片段（如多段"商品正面展示"合并为一条 + 数量）
3. 用结构化表格而非自然语言描述，减少 token 消耗

```
素材概况：共 5 条视频，分割为 32 个片段，可用 24 个。
| 片段ID | 时长 | 内容 | 卖点关联 | 质量 |
| S01   | 3.2s | 商品正面展示 | 高颜值(0.9) | 0.85 |
| S02   | 5.1s | 模特试穿走动 | 轻便(0.8) | 0.78 |
| ...
```

### 4.2 分镜脚本 Schema 设计

LLM 输出的核心数据结构——分镜脚本，必须严格遵守 JSON Schema：

```typescript
interface Storyboard {
  id: string;
  title: string;                    // 脚本标题/主题
  totalDuration: number;            // 目标总时长（秒）
  style: 'energetic' | 'calm' | 'tech' | 'warm' | 'luxury';
  narrative: string;                // 叙事策略简述
  shots: Shot[];                    // 分镜序列
  audio: AudioConfig;               // 音频配置
}

interface Shot {
  index: number;                    // 镜头序号
  description: string;              // 画面描述（供素材匹配用）
  duration: number;                 // 目标时长（秒）
  suggestedShotIds: string[];       // 推荐使用的素材片段 ID
  sellingPoint?: string;            // 该镜头要传达的卖点
  subtitle?: string;                // 叠加字幕文案
  subtitleStyle?: 'pop' | 'minimal' | 'bold';
  transition: TransitionType;       // 与下一镜头的转场
  motionEffect?: 'zoom_in' | 'zoom_out' | 'pan_left' | 'pan_right' | 'none';
}

type TransitionType = 'cut' | 'dissolve' | 'fade_black' | 'slide_left' | 'zoom';

interface AudioConfig {
  bgmStyle: string;                 // BGM 风格标签
  bgmVolume: number;                // 0-1 BGM 音量
  voiceover?: {
    text: string;                   // TTS 配音文案
    tone: 'professional' | 'friendly' | 'energetic';
    speed: number;                  // 语速倍率
  };
  keepOriginalAudio: boolean;       // 是否保留原始音频
  originalAudioVolume?: number;     // 原始音频音量
}
```

### 4.3 多条差异化生成策略

同一批素材生成 M 条视频时，核心难点是**避免雷同**。差异化维度：

| 维度 | 策略 | 示例 |
|------|------|------|
| 叙事顺序 | 不同的故事线排列 | 视频 1: 痛点→解决→展示；视频 2: 展示→细节→场景 |
| 卖点侧重 | 每条突出不同核心卖点 | 视频 1: 主打"轻便"；视频 2: 主打"防水" |
| 节奏风格 | 不同剪辑节奏 | 视频 1: 快节奏短镜头；视频 2: 慢节奏长镜头 |
| 素材组合 | 不同片段选择与排列 | 避免多条视频使用相同的前 3 个镜头 |
| 开头钩子 | 不同的开场方式 | 视频 1: 产品特写开场；视频 2: 场景带入开场 |

**实现方式**：在 Prompt 中明确约束差异化要求，一次 LLM 调用生成全部 M 条脚本（而非 M 次独立调用），使模型能全局感知已有脚本从而避免重复：

```
请根据以上素材和卖点，生成 {M} 条差异化的分镜脚本。
要求：
1. 每条脚本的叙事策略必须不同（开头、结构、节奏）
2. 每条脚本侧重不同的核心卖点
3. 同一素材片段在不同脚本中不能连续出现在相同位置
4. 每条脚本必须严格控制在目标时长 ±2 秒以内
```

### 4.4 Prompt 与 Function Calling 设计

**System Prompt 核心结构**：

```
你是一位专业的视频编导 AI。你的任务是根据用户提供的视频素材分析结果和业务需求，
生成结构化的分镜脚本。

你必须遵守以下规则：
1. 每条脚本必须覆盖至少 80% 的用户卖点
2. 总时长必须在目标时长 ±2 秒以内
3. 只能使用素材库中已有的片段，不能假设不存在的素材
4. 转场选择必须符合上下镜头的内容逻辑
5. 字幕文案必须简洁，单行不超过 15 个字
```

**Function Calling 定义**：

使用 Structured Output / Function Calling 确保 LLM 输出严格符合 Schema，避免自由文本输出导致解析失败：

```typescript
const tools = [{
  type: 'function',
  function: {
    name: 'generate_storyboards',
    description: '根据素材分析和业务需求生成多条差异化分镜脚本',
    parameters: {
      type: 'object',
      properties: {
        storyboards: {
          type: 'array',
          items: { $ref: '#/definitions/Storyboard' },
          description: '分镜脚本数组，数量等于用户要求的生成条数'
        }
      },
      required: ['storyboards']
    }
  }
}];
```

### 4.5 脚本自检与约束校验

LLM 生成的脚本不一定满足所有硬约束，需要后置校验层：

```
生成脚本 → 硬约束校验 → 通过 → 进入编排
                ↓ 不通过
          错误反馈给 LLM → 重新生成（最多 3 轮）
```

校验规则：

| 规则 | 检查逻辑 | 处理方式 |
|------|---------|---------|
| 时长约束 | `sum(shot.duration)` 是否在目标 ±2s 内 | 要求 LLM 调整镜头时长 |
| 素材存在性 | `suggestedShotIds` 是否都在素材库中 | 移除无效引用，要求补充 |
| 卖点覆盖率 | 统计脚本覆盖了多少用户卖点 | 低于 80% 时要求补充 |
| 转场合理性 | 相邻镜头内容是否适合选用的转场 | 自动降级为 `cut` |
| 差异化检查 | M 条脚本间的片段重合度 | 重合度 > 60% 时要求调整 |

## 5. 智能编排引擎

编排引擎负责将抽象的脚本"落地"为可渲染的时间轴。

### 5.1 素材-脚本语义匹配

脚本中的 `suggestedShotIds` 是 LLM 的建议，但可能存在匹配不精确的情况。编排引擎需要二次匹配：

```
脚本镜头描述: "模特在户外展示商品轻便特性"
                    │
                    ▼
          Embedding 向量化
                    │
                    ▼
    与素材库所有 Shot 做相似度检索
                    │
                    ▼
    ┌──────────────────────────┐
    │ Top-K 候选片段            │
    │ S12: 模特户外走动(0.92)   │
    │ S07: 商品展示手持(0.85)   │
    │ S19: 户外场景空镜(0.71)   │
    └──────────────────────────┘
                    │
                    ▼
          结合 LLM 建议 + 质量分 + 已用去重
                    │
                    ▼
              最终选择 S12
```

**去重机制**：已在当前脚本中使用的片段，在后续镜头匹配时降低优先级（而非完全排除，因为关键素材可能需要复用）。

### 5.2 时间轴编排

将匹配后的素材片段按脚本顺序组装为时间轴：

```typescript
interface TimelineTrack {
  type: 'video' | 'audio' | 'subtitle' | 'effect';
  clips: TimelineClip[];
}

interface TimelineClip {
  trackType: string;
  sourceId: string;        // 素材片段 ID 或生成资源 ID
  sourceInPoint: number;   // 素材内起始点
  sourceOutPoint: number;  // 素材内结束点
  timelineStart: number;   // 在时间轴上的起始位置
  timelineDuration: number;
  effects: Effect[];       // 叠加效果（缩放、位移等）
  transition?: {
    type: TransitionType;
    duration: number;      // 转场时长（通常 0.3-1s）
  };
}
```

**时长精确控制**：

素材片段时长与脚本要求时长可能不一致：

| 情况 | 处理策略 |
|------|---------|
| 素材 > 脚本要求 | 智能裁剪：优先保留高质量帧区间 |
| 素材 < 脚本要求 | 速度微调（0.8x-1.0x 慢放）或从候选列表追加片段 |
| 素材 ≈ 脚本要求 | 直接使用，微调 ±0.5s 以适配转场 |

### 5.3 音画同步

音画同步是成片专业感的关键，涉及三层对齐：

```
┌─────────────────────────────────────────────────┐
│ 视频轨  │ Shot1 │ Trans │ Shot2 │ Trans │ Shot3 │
├─────────────────────────────────────────────────┤
│ 配音轨  │ "这款产品..."  │  "采用了..."  │ ...  │
├─────────────────────────────────────────────────┤
│ 字幕轨  │ [这款产品] │  [采用了]  │  [...]     │
├─────────────────────────────────────────────────┤
│ BGM轨   │ ♫ ─────────────────────────────── ♫  │
│ 节拍点  │ ↑     ↑     ↑     ↑     ↑     ↑     │
└─────────────────────────────────────────────────┘
```

**配音时长反算**：

传统流程是先定画面再配音，但 AI 成片需要反过来——先由 TTS 生成配音，再根据配音时长调整镜头时长：

```
脚本文案 → TTS 生成配音 → 获得实际配音时长
                              ↓
                    以配音时长为锚点反算镜头时长
                              ↓
                    调整素材裁剪区间
```

**BGM 节拍对齐**：

通过音频分析提取 BGM 的节拍点（Beat），将镜头切换时机对齐到节拍点附近（±100ms 容差），提升视觉节奏感。

```typescript
function alignToBeats(cuts: number[], beats: number[], toleranceMs = 100): number[] {
  return cuts.map(cut => {
    const nearest = beats.reduce((prev, curr) =>
      Math.abs(curr - cut) < Math.abs(prev - cut) ? curr : prev
    );
    return Math.abs(nearest - cut) <= toleranceMs / 1000 ? nearest : cut;
  });
}
```

### 5.4 AI 增强层

在素材本身之外叠加的 AI 生成内容：

| 增强类型 | 技术方案 | 说明 |
|---------|---------|------|
| 智能配音 | TTS（如 Edge TTS / Fish Audio） | 根据脚本文案生成多音色配音 |
| 智能字幕 | 脚本文案 → SRT 生成 | 按配音时间戳精确对齐字幕 |
| 动态字幕特效 | 关键词高亮 + 打字机效果 | 卖点关键词用不同颜色/大小强调 |
| 封面生成 | 从成片中提取最佳帧 + 文字叠加 | 自动生成视频封面图 |
| 智能补帧 | 视频插帧模型（RIFE） | 素材帧率不足时提升流畅度 |

### 5.5 差异化混剪策略

同素材池生成多条成片的编排层差异化：

```
脚本 1 (快节奏): S01→S05→S12→S03→S18→S07  (平均 2s/镜头)
脚本 2 (慢节奏): S12→S01→S19→S05           (平均 5s/镜头)
脚本 3 (卖点驱动): S03→S07→S12→S18→S01→S05  (按卖点分组)
```

编排引擎确保：
- 同一片段在不同视频中使用不同的裁剪区间
- 转场类型在不同视频间有变化
- BGM 选用不同曲目或同曲目不同段落

## 6. 渲染合成与批量调度

### 6.1 渲染方案选型

| 方案 | 适用场景 | 优势 | 限制 |
|------|---------|------|------|
| **FFmpeg 服务端渲染** | 生产环境批量导出 | 编码质量高、格式兼容性好、支持 GPU 加速 | 需要服务端 GPU 资源 |
| **WebAssembly 浏览器端** | 实时预览、轻量编辑 | 零服务端成本、即时反馈 | 受限于设备性能、长视频编码慢 |
| **混合方案（推荐）** | 预览 + 导出分离 | 兼顾体验和质量 | 架构复杂度高 |

**推荐的混合方案**：

```
┌─────────────────────────────┐
│         浏览器端              │
│                             │
│  Canvas/WebGL 实时预览       │
│  低分辨率 + 关键帧渲染       │  ◄── 用户编辑/调整时
│  WebAssembly 轻量合成        │
└──────────────┬──────────────┘
               │ 确认导出
               ▼
┌─────────────────────────────┐
│         服务端               │
│                             │
│  FFmpeg GPU 编码             │
│  高分辨率 + 全帧渲染         │  ◄── 最终导出时
│  多格式/多分辨率批量输出      │
└─────────────────────────────┘
```

### 6.2 批量任务调度

M 条视频的并行渲染需要调度器管理：

```typescript
interface RenderTask {
  taskId: string;
  storyboardId: string;
  priority: number;           // 优先级（用户指定或按序号）
  status: 'pending' | 'preprocessing' | 'rendering' | 'encoding' | 'completed' | 'failed';
  progress: number;           // 0-100
  retryCount: number;
  outputConfig: OutputConfig;
}

interface BatchScheduler {
  maxConcurrent: number;      // 最大并行渲染数（根据 GPU 数量）
  taskQueue: RenderTask[];
  // 调度策略：优先级队列 + 资源感知
  schedule(): void;
  // 失败重试：指数退避，最多 3 次
  retry(taskId: string): void;
}
```

**调度策略**：

```
任务队列 → 按优先级排序 → 检查可用 GPU/CPU 资源
                              ↓ 有空闲资源
                         分配资源，启动渲染
                              ↓ 资源满
                         等待，监听完成事件
```

### 6.3 多平台输出适配

一次生成适配多个投放平台：

| 平台 | 分辨率 | 比例 | 编码 | 码率 |
|------|-------|------|------|------|
| 抖音/快手 | 1080×1920 | 9:16 | H.264 | 6-8 Mbps |
| 视频号 | 1080×1920 | 9:16 | H.264 | 6-8 Mbps |
| B 站 | 1920×1080 | 16:9 | H.264 | 8-12 Mbps |
| 小红书 | 1080×1080 | 1:1 | H.264 | 6-8 Mbps |
| YouTube | 1920×1080 / 3840×2160 | 16:9 | H.265 | 12-20 Mbps |

**智能裁剪适配**：

同一素材在不同比例下需要智能裁剪，而非简单拉伸：

```
16:9 素材 → 9:16 竖屏输出
                │
   ┌────────────┼─────────────┐
   │            │             │
   ▼            ▼             ▼
 主体居中裁剪  智能追踪裁剪   画中画+模糊背景
 (简单快速)   (检测主体位置   (保留全画面
              动态调整裁剪框)  上下补模糊)
```

## 7. 横切关注点

### 7.1 质量评估体系

成片质量自动评估，用于筛选最佳结果或触发重新生成：

```
┌────────────────────────────────────────────┐
│              成片质量评估                     │
│                                            │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│  │ 技术质量  │ │ 内容质量  │ │ 业务质量  │   │
│  │          │ │          │ │          │   │
│  │ 画面清晰度│ │ 叙事连贯性│ │ 卖点覆盖率│   │
│  │ 音频质量  │ │ 节奏流畅度│ │ 品牌调性  │   │
│  │ 转场平滑度│ │ 字幕可读性│ │ 平台合规  │   │
│  │ 编码质量  │ │ 情感传达  │ │ CTA 有效性│   │
│  └──────────┘ └──────────┘ └──────────┘   │
│                    │                       │
│            综合质量分 (加权)                  │
│           ≥ 阈值 → 通过                     │
│           < 阈值 → 标记/重新生成             │
└────────────────────────────────────────────┘
```

**自动化评估指标**：

| 维度 | 指标 | 评估方式 |
|------|------|---------|
| 技术质量 | VMAF / SSIM 视频质量分 | FFmpeg 计算 |
| 节奏 | 镜头平均时长方差 | 统计分析 |
| 卖点覆盖 | 脚本卖点 vs 用户卖点的匹配数 | 文本匹配 |
| 差异化 | 多条成片间片段重合度 | 集合运算 |
| 音画同步 | 字幕与语音的时间偏差 | ASR + 对齐检测 |

### 7.2 容错与降级

多模型调用链路中任何环节都可能失败：

| 故障点 | 降级策略 |
|-------|---------|
| 素材分析模型不可用 | 降级为基于规则的分析（抽帧 + OpenCV 基础检测） |
| LLM 调用失败 | 重试 → 降级为模板脚本（预设行业模板填充素材） |
| TTS 服务不可用 | 降级为纯 BGM + 字幕模式，跳过配音 |
| BGM 匹配失败 | 使用默认 BGM 库中的通用配乐 |
| 渲染失败 | 降低分辨率/码率重试 → 降级为图片轮播模式 |
| 单条脚本校验失败 | 跳过该条，优先交付可用的成片 |

**关键原则**：宁可交付质量稍低的成片，也不要让用户等待后看到空结果。

### 7.3 性能优化

| 优化点 | 策略 | 效果 |
|-------|------|------|
| 素材分析缓存 | 同一素材的分析结果持久化存储，不重复计算 | 重复使用素材时秒级返回 |
| LLM 流式输出 | 脚本生成后立即开始素材匹配，不等全部脚本完成 | Pipeline 并行，减少总耗时 |
| 预加载 | 在脚本生成阶段预加载高概率使用的素材片段 | 编排阶段零等待 |
| GPU 渲染池 | 维护 GPU 实例池，按需扩缩容 | 避免冷启动 |
| 分段渲染 | 将长视频拆分为 5s 片段并行渲染，最后拼接 | 线性扩展渲染速度 |

**端到端耗时估算**（单条 30s 视频）：

| 阶段 | 耗时 | 是否可并行 |
|------|------|-----------|
| 素材分析 | 10-30s | 多条素材并行 |
| LLM 脚本生成 | 5-15s | M 条一次生成 |
| 素材匹配 & 编排 | 2-5s | 与脚本生成流式衔接 |
| TTS 配音生成 | 3-8s | 与编排并行 |
| 渲染合成 | 15-45s | M 条并行 |
| **总计** | **~30-90s** | 优化后 |

### 7.4 成本控制

多模型调用链路的 token 和算力成本管理：

| 成本项 | 优化策略 |
|-------|---------|
| LLM Token | 素材摘要压缩、一次调用生成多条脚本、缓存相似请求 |
| 视觉模型 | 仅对高质量帧调用识别模型，低质量帧跳过 |
| TTS | 短文案合并调用，缓存常用话术的音频 |
| GPU 渲染 | 非高峰时段执行批量任务，使用竞价实例 |
| 存储 | 中间产物（抽帧图片）设置 TTL 自动清理 |
