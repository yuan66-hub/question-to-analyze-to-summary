# Skills 蒸馏模型：基于 Prompt-time 的知识蒸馏策略

## 核心概念

Skills 蒸馏不是传统的模型蒸馏（teacher → student 权重迁移），而是 **Prompt-time Knowledge Distillation**：将专家知识编码为结构化的 Skill 文件，在推理时注入模型 Context，使任意模型获得专家级能力。

| 对比维度 | 传统蒸馏 | Skills 蒸馏 |
|----------|---------|------------|
| 载体 | 模型权重 | Skill 文件（Markdown） |
| 方式 | 大模型训练小模型 | 专家经验编码为结构化 Prompt |
| 成本 | 需要训练数据和 GPU 算力 | 只需编写结构化文档 |
| 生效时间 | 训练后生效 | 即时注入，即时生效 |
| 迭代方式 | 重新训练 | 修改文件，下次会话生效 |
| 模型变化 | 权重改变 | 权重不变，Context 增强 |

## 蒸馏层次模型

```
┌──────────────────────────────────────────────┐
│  Layer 3: 工作流蒸馏 (Workflow Distillation)   │
│  TDD、Debugging、Code Review、Brainstorming   │
│  → 蒸馏的是「做事方法论」                       │
├──────────────────────────────────────────────┤
│  Layer 2: 领域知识蒸馏 (Domain Distillation)   │
│  Vue 性能优化、NestJS 最佳实践、React 模式      │
│  → 蒸馏的是「专家级技术知识」                    │
├──────────────────────────────────────────────┤
│  Layer 1: 约束蒸馏 (Constraint Distillation)   │
│  Git 规范、代码风格、安全规则                    │
│  → 蒸馏的是「行为规范和偏好」                    │
└──────────────────────────────────────────────┘
```

- **Layer 1** 通过 CLAUDE.md / rules 实现，定义模型的行为边界
- **Layer 2** 通过 Domain Skills 实现，注入特定技术领域的深度知识
- **Layer 3** 通过 Workflow Skills 实现，强制模型按严格流程执行

## 三种蒸馏模式

### 模式 1: 专家知识 → Skill

将技术专家的经验、演讲、论文蒸馏为结构化 Skill。

**示例**: vue2-performance Skill
- 来源：Guillaume Chau 在 Vue Conf US 2019 的 9 个性能秘密
- 蒸馏方式：症状-方案决策表 + 代码模板 + 使用条件
- 效果：Haiku + Skill ≈ Vue 性能专家

**蒸馏公式**:

```
专家知识 = 决策树 + 代码模板 + 反模式 + 检查清单
Skill = 结构化(专家知识)
小模型 + Skill ≈ 大模型的专业能力
```

### 模式 2: 工作流程 → Skill

将软件工程方法论蒸馏为强制执行的流程。

**示例**: TDD Skill
- 来源：测试驱动开发方法论
- 蒸馏方式：RED → GREEN → REFACTOR 严格步骤 + 检查清单
- 效果：强制模型按纪律执行，消除"随意编码"

### 模式 3: 模型分层 + Skill 组合

不同能力的模型搭配不同粒度的 Skill，实现成本最优。

```
┌─────────────┬──────────────────┬──────────────────┐
│ 模型层级     │ 职责              │ Skill 配置        │
├─────────────┼──────────────────┼──────────────────┤
│ Opus (推理)  │ 架构决策、复杂分析  │ 无需 Skill，靠推理 │
│ Sonnet (编码)│ 主要开发工作       │ Domain + Workflow │
│ Haiku (轻量) │ Worker Agent      │ Focused Skill     │
└─────────────┴──────────────────┴──────────────────┘
```

**关键洞察**: Haiku + 高质量 Skill ≈ Sonnet 裸跑，成本降低 3 倍。

## 高质量 Skill 的结构化模板

### 可蒸馏知识的特征

- 有明确的**决策树**（if X then Y）
- 有**检查清单**（不遗漏关键步骤）
- 有**代码模板**（减少模型生成变异）
- 有**反模式**（明确告诉模型什么不该做）

### Skill 文件标准结构

```markdown
---
name: <skill-name>
description: "<触发条件描述，说明何时应激活此 Skill>"
---

# <Skill 名称>

## 决策表
| 场景/症状 | 推荐方案 | 参考 |
|-----------|---------|------|
| ...       | ...     | #N   |

## 核心流程
1. 步骤一（明确的动作）
2. 步骤二（明确的动作）
3. 验证步骤（确认结果）

## 代码模板
// 标准实现模式，减少模型输出变异

## 反模式
- ❌ 不要做 X（解释原因）
- ❌ 避免 Y（解释后果）

## 检查清单
- [ ] 条件 A 已满足
- [ ] 条件 B 已验证
- [ ] 无已知反模式
```

### 蒸馏质量评估标准

| 维度 | 低质量 Skill | 高质量 Skill |
|------|-------------|-------------|
| 决策 | "视情况而定" | 明确的 if-then 决策表 |
| 步骤 | 模糊描述 | 可机械执行的具体步骤 |
| 代码 | 无模板 | 提供标准代码模板 |
| 边界 | 未说明 | 明确列出反模式和禁止项 |
| 验证 | 无 | 提供检查清单和完成条件 |

## 与传统蒸馏技术的互补关系

Skills 蒸馏和传统模型蒸馏并非替代关系，而是互补：

```
┌─────────────────────────────────────────────────────┐
│              模型能力增强的完整光谱                     │
│                                                     │
│  Pre-training → Fine-tuning → RLHF → Skills 蒸馏    │
│  (基础能力)    (领域适配)    (对齐)   (即时专家化)     │
│                                                     │
│  ←── 需要算力，一次性 ────→ ←── 零算力，可迭代 ──→    │
└─────────────────────────────────────────────────────┘
```

- **传统蒸馏**适合：通用能力迁移、模型压缩、延迟优化
- **Skills 蒸馏**适合：特定领域专家化、工作流标准化、快速迭代

## 实际应用场景

### 场景 1: 团队知识沉淀

```
资深工程师的 Code Review 经验
    ↓ 蒸馏为
code-review Skill（检查项 + 严重级别 + 修复建议模板）
    ↓ 效果
初级工程师 + AI + Skill = 接近资深工程师的 Review 能力
```

### 场景 2: 技术栈迁移

```
旧框架的最佳实践文档
    ↓ 蒸馏为
migration Skill（映射表 + 迁移步骤 + 常见陷阱）
    ↓ 效果
AI 按标准化流程执行迁移，减少人工判断
```

### 场景 3: 成本优化

```
原方案: Opus 处理所有任务（高成本）
    ↓ 蒸馏关键知识为 Skills
新方案: Haiku/Sonnet + Skills 处理 80% 任务（成本降 3-5x）
        Opus 仅处理真正需要深度推理的 20%
```

## 总结

Skills 蒸馏的本质是 **将专家知识从"人脑/大模型"迁移到"结构化文档"，再通过 Context 注入让任意模型获得该能力**。它是一种零训练成本、即时生效、可持续迭代的知识蒸馏方式，与传统模型蒸馏形成互补，特别适合：

1. **快速专家化** — 无需训练即可让模型具备特定领域能力
2. **成本优化** — 用小模型 + Skill 替代大模型裸跑
3. **知识沉淀** — 将团队经验标准化为可复用的 Skill 资产
4. **质量保障** — 通过检查清单和流程约束消除模型的随意性
