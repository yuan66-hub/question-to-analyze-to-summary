# 多视频片段合成、字幕生成与音频添加的 FFmpeg 流水线设计

## 1. 场景定义

目标场景是：

- 输入多个视频片段
- 先合成为一个完整视频
- 根据最终视频的目标时长生成字幕时间轴
- 再为视频添加配音、BGM 或混音
- 最终输出一个可播放的视频文件

这个场景里，`ffmpeg` 很适合做三件事：

- 视频拼接
- 字幕挂载或烧录
- 音频替换、裁剪、循环、混音

但有一点要先说明：

`ffmpeg` 本身不负责“智能生成字幕内容”或“理解文案后自动分配字幕语义”，它更适合处理已经准备好的字幕文件，比如 `srt` 或 `ass`。  
所以“根据指定时长生成字幕”通常应该拆成两步：

1. 业务层先根据目标时长生成 `srt/ass`
2. `ffmpeg` 再把字幕挂到视频里，或者直接烧录进画面

## 2. 最重要的结论

如果你的目标是“更高效、合成更快”，推荐的优先级如下：

1. **先尽量完成视频拼接，拿到最终时间轴**
2. **再基于最终时长生成字幕文件**
3. **最后一次性加音频并输出**

原因很简单：

- 字幕时间轴依赖最终视频时长，太早生成容易失准
- 音频长度通常也依赖最终成片长度，尤其是配音和 BGM 循环/裁剪
- 真正耗时的通常不是“加一个音频流”，而是**视频重编码**
- 只要发生“烧录字幕”“缩放”“转码”“滤镜链”，大概率就要重新编码视频

所以速度优化的核心不是“把所有事塞进一条命令”，而是：

**尽量避免重复编码，最好只在最后一步编码一次。**

## 3. 哪些操作最耗时

按常见成本从高到低大致可以理解为：

1. **硬字幕烧录**：最贵，因为视频帧要逐帧重新渲染
2. **缩放、裁剪、转帧率、滤镜链**：也会触发重编码
3. **`concat filter` 拼接**：灵活，但通常比直接流复制更慢
4. **音频转码/混音**：有开销，但通常比视频重编码轻很多
5. **流复制拼接和封装**：最快，几乎只是重组容器

因此，最值得优先优化的部分是：

- 尽量让视频片段参数一致，争取使用 `concat demuxer + -c copy`
- 能用软字幕就不要烧录硬字幕
- 如果必须烧录字幕，把烧录放在最后一步，只做一次
- 如果音频只是替换或外挂，不要提前多轮导出中间视频

## 4. 推荐处理顺序

### 方案 A：速度优先，推荐

这是大多数生产场景更合适的顺序：

1. 先检查所有视频片段是否同编码、同分辨率、同帧率、同音频参数
2. 如果一致，先直接拼接成一个中间视频，尽量 `-c copy`
3. 根据拼接后的总时长，业务层生成字幕文件 `srt/ass`
4. 最后一步再统一加音频和字幕输出

这个方案的优点：

- 拼接最快
- 时间轴最稳定
- 最终只需要一次重编码，或者甚至不需要视频重编码
- 出问题时好排查

### 方案 B：一条命令全部做完

可以做到，但前提是：

- 你已经提前准备好了字幕文件
- 你清楚片段是否能直接拼接
- 你接受命令会更长、更难维护

这个方案适合固定模板任务，不太适合复杂业务流程。

### 方案 C：先加字幕再拼接

通常不推荐。

因为：

- 每段视频都要单独处理字幕，容易多次编码
- 最终拼接后字幕节奏可能还要重新调整
- 总体耗时更长

## 5. 两种拼接方式怎么选

### 5.1 `concat demuxer`：最快

适用条件：

- 所有片段编码格式一致
- 分辨率一致
- 帧率一致
- 音频采样率/声道一致

优点：

- 速度最快
- 可以直接 `-c copy`
- 几乎不做重新编码

示例文件 `list.txt`：

```txt
file 'part1.mp4'
file 'part2.mp4'
file 'part3.mp4'
```

快速拼接：

```bash
ffmpeg -f concat -safe 0 -i list.txt -c copy merged.mp4
```

### 5.2 `concat filter`：最灵活

适用条件：

- 不同片段分辨率不同
- 帧率不同
- 编码格式不同
- 有的片段没音频，有的有音频

优点是兼容性强，缺点是通常要重编码，速度更慢。

下面这个示例默认每个片段都同时有视频和音频流。  
如果你的真实数据里“有的片段没音频，有的有音频”，通常要先给无音频片段补静音轨，或者把视频拼接和音频处理拆开做。

示例：

```bash
ffmpeg -i part1.mp4 -i part2.mp4 -i part3.mp4 \
  -filter_complex "[0:v][0:a][1:v][1:a][2:v][2:a]concat=n=3:v=1:a=1[v][a]" \
  -map "[v]" -map "[a]" \
  -c:v libx264 -preset veryfast -crf 23 \
  -c:a aac -b:a 192k \
  merged.mp4
```

如果你追求吞吐，最好的工程策略不是每次在线用 `concat filter` 救火，而是：

**在素材入库阶段先标准化编码参数，后续在线请求就能直接走 `concat demuxer`。**

## 6. 字幕应该什么时候生成

如果字幕是“根据最终成片时长分配每句出现时间”，最合理的时机是：

**先拿到合成后的视频总时长，再生成字幕文件。**

比如有三句文案：

- 第一句
- 第二句
- 第三句

最终成片时长是 `30s`，业务层可以按规则分配时间：

- 平均分配
- 按字数占比分配
- 按语音时长占比分配
- 按镜头片段边界分配

然后生成 `subs.srt` 或 `subs.ass`。

举例，一个简单的 `srt`：

```srt
1
00:00:00,000 --> 00:00:08,000
第一句字幕

2
00:00:08,000 --> 00:00:18,000
第二句字幕

3
00:00:18,000 --> 00:00:30,000
第三句字幕
```

这里最关键的工程判断是：

- **字幕内容生成**：交给业务层、模型层或脚本层
- **字幕应用到视频**：交给 `ffmpeg`

不要把“字幕内容生成”和“字幕渲染”混为一件事。

## 7. 硬字幕和软字幕怎么选

### 7.1 软字幕：更快

软字幕的意思是字幕作为独立字幕流封装进视频，不直接烧到画面里。

优点：

- 通常不需要重编码视频
- 速度快
- 后续可以切换开关
- 修改字幕成本低

缺点：

- 某些平台上传后不一定保留字幕流
- 短视频平台通常更偏向直接画面可见的硬字幕

### 7.2 硬字幕：更通用，但更慢

硬字幕就是把字幕直接画到视频画面上。

优点：

- 所见即所得
- 平台兼容性最好
- 上传后不会丢字幕

缺点：

- 必须重编码视频
- 最耗时

如果你是做抖音、快手、视频号这类短视频成片输出，通常还是**硬字幕更实用**。  
但如果是在内部预览、审核、素材流转环节，**软字幕更高效**。

## 8. 一条命令完成的几种写法

### 8.1 同参数片段 + 硬字幕 + BGM

这是比较常见的一条命令方案：

```bash
ffmpeg -f concat -safe 0 -i list.txt \
  -stream_loop -1 -i bgm.mp3 \
  -vf "subtitles=subs.srt:force_style='FontName=Microsoft YaHei,FontSize=18,Outline=1,Shadow=0'" \
  -map 0:v:0 -map 1:a:0 \
  -c:v libx264 -preset veryfast -crf 23 -pix_fmt yuv420p \
  -c:a aac -b:a 192k \
  -t 30 \
  output.mp4
```

说明：

- `-f concat -safe 0 -i list.txt`：读取片段列表
- `-stream_loop -1 -i bgm.mp3`：BGM 不够长时循环
- `-vf subtitles=...`：烧录硬字幕
- `-map 0:v:0 -map 1:a:0`：取拼接后视频和 BGM 音频
- `-t 30`：输出目标时长 30 秒

这个命令可以“一次出成片”，但因为有 `subtitles` 过滤器，视频一定会重编码。

### 8.2 同参数片段 + 软字幕 + 音频

如果只是想更快出片，可以用软字幕：

```bash
ffmpeg -f concat -safe 0 -i list.txt \
  -stream_loop -1 -i bgm.mp3 \
  -i subs.srt \
  -map 0:v:0 -map 1:a:0 -map 2:0 \
  -c:v copy \
  -c:a aac -b:a 192k \
  -c:s mov_text \
  -t 30 \
  output.mp4
```

这个方案的优势是：

- 视频不重编码
- 速度明显更快

但它不是烧录字幕，而是把字幕作为字幕轨封装进去。

### 8.3 片段参数不一致 + 拼接 + 字幕 + 混音

如果片段参数不一致，又要配音和 BGM 混音，可以用一条更完整的命令：

下面这个例子同样假设每个输入片段都带有音频流：

```bash
ffmpeg -i part1.mp4 -i part2.mp4 -i part3.mp4 -stream_loop -1 -i bgm.mp3 -i voice.mp3 \
  -filter_complex "[0:v][0:a][1:v][1:a][2:v][2:a]concat=n=3:v=1:a=1[v][basea]; \
                   [3:a]atrim=0:30,volume=0.25[bgm]; \
                   [4:a]volume=1.2[voice]; \
                   [basea][bgm][voice]amix=inputs=3:duration=first[aout]" \
  -map "[v]" -map "[aout]" \
  -vf "subtitles=subs.srt" \
  -c:v libx264 -preset veryfast -crf 23 -pix_fmt yuv420p \
  -c:a aac -b:a 192k \
  -t 30 \
  output.mp4
```

这个方案最灵活，但也最重。

如果你每天要大量批处理，这种“一条超长命令”通常不是最优解，维护成本高，吞吐也不一定最好。

## 9. 更高效的工程化方案

如果你想真正提升整体生成速度，建议不要只盯着 `ffmpeg` 命令本身，而是优化整条流水线。

推荐顺序如下：

### 第一步：素材标准化

在素材入库阶段就把所有片段转成统一规格，例如：

- `1920x1080`
- `25fps`
- `h264 + aac`
- 音频采样率 `44100` 或 `48000`

这样在线请求时就可以直接拼接，不需要临时做复杂兼容处理。

### 第二步：快速拼接

如果规格一致，直接：

```bash
ffmpeg -f concat -safe 0 -i list.txt -c copy merged.mp4
```

这一步是最快的。

### 第三步：根据最终时长生成字幕文件

业务层根据 `merged.mp4` 的总时长，生成 `subs.srt` 或 `subs.ass`。

如果你只是想让字幕按时长均匀分布，这一步完全不该交给 `ffmpeg`，而应该在上层逻辑里做。

### 第四步：最终导出

按目标选择两种模式：

- 如果要最快：挂软字幕，视频尽量 `copy`
- 如果要最终发布：烧录硬字幕，并在这一步一次性加入音频

换句话说：

**真正高效的做法不是“所有步骤同时做”，而是“前面尽量不编码，最后只编码一次”。**

## 10. 如果只能优化一个环节，优先优化哪里

如果现在只能选一个点做优化，优先级建议是：

### 第一优先级：避免重复视频编码

最值钱的优化。

例如不要这样：

1. 每个片段先烧字幕导出一次
2. 再拼接导出一次
3. 再加音频导出一次

这样会让视频被重复编码很多次。

应该尽量改成：

1. 先拼接
2. 生成字幕文件
3. 最后一次性加字幕、加音频、导出

### 第二优先级：尽量使用 `concat demuxer`

只要素材参数能统一，这一步能节省很多时间。

### 第三优先级：能用软字幕时就别烧录

如果业务允许，软字幕能明显提升速度。

### 第四优先级：把字幕生成放在业务层

字幕文本和时间轴分配，尽量在上层逻辑中提前算好，不要在媒体处理层里“边试边调”。

## 11. 实战建议

如果你的目标是“生成速度更快”，建议优先采用下面这套组合：

### 发布型场景

- 片段先标准化
- 使用 `concat demuxer` 快速拼接
- 根据最终时长生成 `srt/ass`
- 最后一条命令烧录字幕并加音频

特点：

- 成片稳定
- 速度较快
- 适合最终发布

### 预览型场景

- 片段先标准化
- 使用 `concat demuxer` 快速拼接
- 生成 `srt`
- 用软字幕方式封装

特点：

- 速度最快
- 方便修改字幕
- 适合审核和内部流转

## 12. 最终结论

对于“多个视频片段合成，然后根据合成视频指定时长生成字幕，再添加音频”的场景，最优先推荐的思路是：

**先拼接视频，拿到最终时长；再生成字幕文件；最后一次性加音频并导出。**

如果从效率角度排序：

1. **先优化拼接方式**，尽量让素材支持 `concat demuxer`
2. **把字幕生成放到业务层**，不要依赖 `ffmpeg` 生成字幕内容
3. **最终只做一次视频编码**
4. **业务允许的话优先软字幕**

一句话总结：

**`ffmpeg` 可以一条命令做完，但真正更快的不是“命令更长”，而是“尽量少转码，尤其少做视频重编码”。**
