# Vue 3 Reactivity vs Alien Signals

## 一句话总结

Vue 3 早期响应式系统的核心优势是“通用、易用、默认深度响应”；Alien Signals 的核心优势是“更细粒度、传播链路更轻、为无 VDOM/编译期优化铺路”。前者更偏工程易用性，后者更偏极致性能。

---

## 1. 背景脉络

Vue 响应式系统的演进可以粗略分成三段：

1. Vue 2：基于 `Object.defineProperty`
2. Vue 3 早期：基于 `Proxy` 的深度响应式系统
3. Vue 后续探索：引入更接近 `signals` 思路的底层优化，如 Alien Signals，并进一步服务于 Vapor Mode

这篇文章真正想说明的不是“Vue 3 不行”，而是：

- Vue 3 早期响应式已经比 Vue 2 先进很多
- 但在大状态、深层对象、高频更新、复杂组件树场景下，默认深度响应式仍然存在额外成本
- Alien Signals 代表 Vue 团队对“更细粒度、更低运行时开销”方向的继续推进

---

## 2. Vue 3 早期响应式系统在做什么

Vue 3 早期响应式系统可以概括为：

- 用 `Proxy` 拦截对象属性访问和修改
- 用 `effect` 包裹依赖响应式数据的逻辑
- 在 `get` 时 `track`
- 在 `set` 时 `trigger`

可以简化理解成：

```text
读取属性 -> 收集“谁依赖了我”
修改属性 -> 通知“依赖我的人重新运行”
```

### 核心机制

| 机制 | 作用 |
|------|------|
| `reactive()` | 把对象包装成响应式代理 |
| `Proxy.get` | 读取属性时收集依赖 |
| `Proxy.set` | 修改属性时触发依赖 |
| `effect()` | 把副作用逻辑注册成响应式订阅者 |
| `track()` | 建立属性 -> effect 的依赖关系 |
| `trigger()` | 找到相关 effect 并重新调度 |

### 这种设计的优势

- API 自然，符合“直接改对象也能更新 UI”的直觉
- 对对象新增/删除属性、数组操作支持比 Vue 2 更完整
- 对大多数中小型应用来说足够通用
- 开发者不需要一开始就显式声明每一个状态节点

### 这种设计的成本

- 默认是深度响应式，嵌套属性访问也会进入代理逻辑
- 依赖收集发生在运行时，成本会随着访问量累积
- 一旦组件 render 依赖很多属性，更新时容易把较大渲染单元重新跑一遍
- 如果最终仍然走 VDOM diff，那么响应式触发之后还有一层渲染和比对成本

---

## 3. Vue 3 早期方案的性能瓶颈在哪里

文章里提到的重点，不是单次 `get/set` 很慢，而是**大量访问叠加后的总体成本**。

典型瓶颈主要有三类：

### 1. 深层对象访问成本

如果一个大对象有很多层嵌套，并且模板、计算属性或渲染函数频繁读取这些字段，那么每次读取都可能经过代理拦截和依赖追踪。

```text
大对象
  -> 深层嵌套
  -> 高频读取
  -> 大量 track
  -> 额外运行时成本
```

### 2. 更新粒度不够细

虽然 Vue 3 的依赖追踪已经比“整棵树刷新”细很多，但在组件层面，很多时候触发的仍是组件 render 重新执行，而不是像某些 signal 系统那样直接精确更新到更小的计算节点或 DOM 绑定点。

### 3. VDOM 仍有固有成本

响应式系统只负责“知道谁变了”，但 Vue 传统渲染链路仍通常要经过：

```text
状态变化 -> render 重新执行 -> 生成新 VDOM -> diff -> patch DOM
```

所以真实开销往往不只在响应式本身，也在后续渲染链路。

---

## 4. Alien Signals 优化了什么

Alien Signals 可以理解成：Vue 团队对底层响应式引擎做的一次更激进的性能探索。

它的关键点不是单纯把 `Proxy` 替换掉，而是：

- 更强调 `signal / computed / effect` 这类显式响应式原语
- 更强调细粒度依赖关系
- 更强调惰性求值和最小传播
- 更强调底层数据结构与传播算法本身的效率

### 核心思路

| 思路 | 含义 |
|------|------|
| `signal` | 表示一个可变状态源 |
| `computed` | 表示由其他信号推导出的值 |
| `effect` | 表示依赖信号变化后要执行的副作用 |
| Push-Pull | 先推送脏标记，再按需拉取计算 |
| Fine-grained tracking | 更细粒度地追踪“谁依赖谁” |

### Push-Pull 的直观理解

它不是“变化了就立刻把所有下游都算一遍”，而更像：

```text
signal 变化
-> 先把下游标记为 dirty
-> 真正读取 computed 或执行 effect 时
-> 再决定是否重新计算
```

这样做的好处是：

- 避免无意义的立刻重算
- 多次连续更新时更容易合并成本
- 只有真正被消费的值才支付计算代价

---

## 5. Alien Signals 为什么可能更快

文章里强调的提升，主要来自以下几个方面。

### 1. 更细粒度的依赖传播

Vue 传统响应式经常以“组件 render”作为一个较大的更新单元；Signals 更适合把依赖关系收缩到具体状态节点、计算节点、副作用节点。

这意味着：

- 变更传播路径更短
- 无关节点更少被波及
- 更容易做到最小更新

### 2. 更轻的数据结构

Alien Signals 在核心传播中刻意避免依赖通用集合结构，转而使用链表、状态位等更贴近底层性能的实现。

这背后的目的通常是：

- 降低内存分配和 GC 压力
- 降低频繁插入/删除订阅关系的成本
- 提高传播时的遍历效率

### 3. 避免递归传播

它的传播逻辑更偏迭代式而不是递归式，目的是：

- 避免深依赖链导致的调用栈问题
- 减少函数递归带来的额外开销
- 让底层传播过程更稳定可控

### 4. 位标志管理状态

用位运算管理 dirty、tracking、notified 等状态，相比多个对象字段或多个布尔值，更省内存也更快。

### 5. 为 Vapor Mode 提供基础

这点非常重要。Alien Signals 的意义不只是“把响应式再快一点”，而是给未来更激进的渲染优化提供支撑。

如果没有足够细粒度、足够低成本的依赖追踪系统，就很难把 Vue 推向：

- 更少的组件级重渲染
- 更少的 VDOM 参与
- 更接近编译产物直接驱动 DOM 更新

## 6. Vue 3 Proxy 响应式 vs Alien Signals 本质差异

| 维度 | Vue 3 早期 `Proxy` 响应式 | Alien Signals |
|------|---------------------------|---------------|
| 默认模型 | 对象代理、深度响应式 | 显式 signal 图 |
| 开发体验 | 更自然，改对象即可 | 更偏显式状态原语 |
| 依赖粒度 | 属性级追踪，但经常落到组件 render 单元 | 更容易收缩到计算节点/副作用节点 |
| 计算策略 | 运行时追踪 + 触发调度 | 推拉结合，惰性更新更强 |
| 数据结构 | 常规依赖表结构 | 更偏底层定制结构，如链表与状态位 |
| 性能目标 | 通用、稳健、好用 | 极致传播效率、最小更新 |
| 对未来架构意义 | 支撑 Vue 3 常规渲染模型 | 为 Vapor Mode 这类方向铺路 |

一句话理解：

```text
Proxy 响应式偏“通用工程解”
Alien Signals 偏“极限性能解”
```

## 7. 这篇文章还能补充的细节

如果把这篇文章扩展成更完整的技术分析，还可以补下面这些点。

### 1. `Proxy` 和 `signals` 不是简单替代关系

很多人会误解成“signals 一定全方位优于 Proxy”。其实不是。

- `Proxy` 的优点是对对象模型友好，迁移与使用成本低
- `signals` 的优点是状态边界更明确，细粒度优化空间更大
- 两者代表的是不同设计取舍，不只是性能高低之分

### 2. 真正瓶颈常常是“响应式 + 渲染”组合

只讨论响应式层还不够。很多页面卡顿，并不是单纯 `track/trigger` 慢，而是：

- 触发了大组件重渲染
- 触发了大量列表 diff
- 触发了昂贵的计算属性或派生逻辑
- 触发了布局抖动或 DOM 回流

所以优化时要把“状态传播”和“渲染成本”一起看。

### 3. 细粒度系统更依赖编译器或显式建模

Signals 的优势通常要和以下能力一起看才更明显：

- 编译器知道依赖边界
- 模板编译结果足够细
- 状态设计本身就按最小粒度拆分

如果业务代码仍然把很多状态塞在一个大对象里，即使底层是 signals，也未必能自动获得理想收益。

### 4. 开发者心智模型会变化

从“改一个大 reactive 对象”转向“拆成多个 signal”，本质上是在要求开发者更主动地设计状态图。

这会带来两个结果：

- 好处：性能边界更清晰
- 成本：抽象方式更显式，对架构能力要求更高

## 8. 性能优化的典型应用场景

这一部分是最值得带到实战里的。

### 场景 1：超大表格 / 树形表格 / 数据看板

特点：

- 单页几十到几百个可见节点
- 每个节点都绑定多列数据
- 局部单元格更新频繁

这类场景里，如果状态放在一个深层大对象中，默认深度响应式会让访问和传播成本不断累积。

优化思路：

- 用 `shallowRef()` 持有大数据集
- 把高频变化字段拆成更小的独立状态
- 列表渲染做虚拟滚动
- 派生数据尽量缓存，不要每次 render 现算

### 场景 2：实时协作 / 高频流式更新

例如：

- 在线文档协作
- 聊天流式输出
- WebSocket 实时行情
- 拖拽编辑器实时预览

特点是状态更新频率高，若每次更新都触发大范围 render，就会很容易出现卡顿。

优化思路：

- 把高频流式状态与低频页面状态分离
- 仅对真正变化的局部订阅
- 批量合并更新，避免每条消息都引发完整链路刷新
- 对大对象使用浅响应式或原始对象承载

### 场景 3：复杂表单构建器

例如低代码表单、问卷搭建器、规则配置器。

问题通常不是字段少，而是：

- 字段之间有依赖关系
- 联动规则多
- 派生状态多
- 局部编辑很频繁

优化思路：

- 表单 schema 与实时编辑态拆开
- 静态 schema 用 `markRaw()` 或浅响应式
- 高频编辑值单独建模
- 让校验和联动只订阅必要字段

### 场景 4：大型只读配置对象

例如：

- 路由配置树
- 菜单权限树
- 国际化字典
- 图表配置模板

这类数据往往大、层级深、运行时几乎不变。如果仍然做深度响应式，收益很低，成本却真实存在。

优化思路：

- 用 `markRaw()` 标记不需要响应式的数据
- 或用 `shallowRef()` 持有整个配置对象
- 只对少量会变化的控制位做响应式

### 场景 5：图编辑器 / 流程编排器 / 画布类应用

这类应用很接近信号系统的优势场景：

- 节点多
- 连线多
- 局部状态变化频繁
- 需要精确更新局部视图

如果每次拖动节点都触发较大组件树重渲染，性能很容易恶化。

优化思路：

- 节点状态按最小单元拆分
- 将拖拽中的瞬时状态和最终持久状态拆分
- 对画布交互层减少深层 reactive 包装
- 让更新粒度尽量接近单节点或单边

## 9. 在 Vue 项目里怎么落地这些优化

即使你没有直接使用 Alien Signals，也可以在 Vue 3 项目里借鉴这套思路。

### 1. 不要默认把所有东西都塞进一个 `reactive({})`

错误倾向：

```text
一个巨大 reactive 对象
-> 页面所有模块都从里面取值
-> 依赖边界不清晰
-> 后期很难定位性能问题
```

更好的方式：

- 按业务域拆状态
- 按更新频率拆状态
- 按是否需要深度响应拆状态

### 2. 大对象优先考虑浅响应式

适合用 `shallowRef()` / `shallowReactive()` 的情况：

- 数据体积很大
- 深层字段不需要逐层追踪
- 更常见的操作是整体替换，而不是深层局部改写

### 3. 明确标记“不需要响应式”的数据

适合用 `markRaw()` 的情况：

- 第三方类实例
- 地图、图表、编辑器对象
- 大型静态配置
- 不希望被代理污染的数据结构

### 4. 缩小高频更新影响面

把这些状态从主响应式树里拆出来：

- 鼠标位置
- 拖拽过程中的临时值
- 输入中的即时值
- 流式消息缓冲区

因为这些状态更新过于频繁，不适合挂在大组件共享状态上。

### 5. 关注“派生状态”的成本

如果一个 `computed` 很重，而且依赖很多数据，那么即使底层响应式很快，整体仍然可能慢。

需要重点看：

- `computed` 是否做了大循环
- 是否在模板里重复读取昂贵派生值
- 是否可以拆成多个更小的 `computed`
- 是否能提前做索引或缓存

## 10. 面试可复述版

如果面试官问“Vue 3 响应式和 Alien Signals 的区别”，可以这样回答：

> Vue 3 早期响应式系统主要基于 `Proxy`，优势是默认深度响应、开发体验自然；但在大对象、深层嵌套和高频更新场景下，会有一定运行时追踪成本，而且更新后通常还要经过组件 render 和 VDOM diff。Alien Signals 代表的是更细粒度的 signal 思路，通过 push-pull、链表、状态位、惰性计算等方式降低依赖传播成本，并为 Vapor Mode 这种更少依赖 VDOM 的方向提供基础。它不只是“更快一点”，而是响应式架构粒度更细、性能目标更激进。

如果再追问“实际项目怎么优化”，可以继续补：

> 实战里不一定要直接上 signals，但要借鉴它的思路，比如拆小状态、减少深度响应、对大对象用 `shallowRef`、对静态配置用 `markRaw`、把高频更新状态从主响应式树里拆出去。

---

## 11. 最后的判断

这篇文章最值得记住的不是某个具体实现细节，而是一个工程趋势：

- 第一阶段：让“状态变化能驱动 UI”即可
- 第二阶段：让“状态变化尽量少影响无关部分”
- 第三阶段：让“编译结果直接绑定到足够细的响应式单元”

Alien Signals 的价值，就在于它体现了 Vue 从“通用响应式”继续走向“细粒度高性能响应式”的方向。
